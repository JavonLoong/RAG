## "向量化"到底在干什么？

```
  普通搜索（关键词匹配）：                我们做的（语义搜索）：
  
  搜"涡轮疲劳" ──→ 只能找到               搜"涡轮疲劳" ──→ 也能找到
  包含这四个字的文档                       "叶片循环载荷导致的寿命衰减"
                                           （意思一样，但用词不同）
  
  ┌──────────────────────────────────────────────────────┐
  │  怎么做到的？                                        │
  │                                                      │
  │  "涡轮疲劳"     ──→  [0.12, 0.85, -0.33, ...]      │
  │  "叶片寿命衰减"  ──→  [0.11, 0.82, -0.31, ...]      │
  │                        ↑                              │
  │              数字很接近 = 意思相近！                  │
  │                                                      │
  │  这个"文字变数字"的过程就叫 【向量化】               │
  │  干这活的工具叫 bge-m3                               │
  └──────────────────────────────────────────────────────┘
```

## 下午怎么发言

### 1️⃣ 开场——证明你看过材料

> "我看了技术文档，目前数据识别已经完成、产出了 JSON 文件。我们下一步的重点就是 **JSON 汇聚、向量化、入 Chroma**。"

### 2️⃣ 主动提问——显得你在深入思考（挑1-2个问）

> "我想先确认下，JSON 文件的格式是什么样的？一个文件对应一个文档还是一个页面？字段结构能发个样例吗？"

> "分块策略大家怎么考虑的？按固定 512 token 切，还是按文档自然段落切？"

> "bge-m3 先用 API 调用还是直接本地跑？文件量大的话需要考虑批处理。"

### 3️⃣ 表态——领任务

> "第一步的代码实现我来跟：写个 Python 脚本，读 JSON → 分块 → 调 bge-m3 → 写 Chroma，先跑通 demo 再调优。"

### 4️⃣ 问后续——展示全局思考

> "第一步跑通之后，检索那边是直接用 Chroma 的内置相似度搜索，还是要加重排序？"
