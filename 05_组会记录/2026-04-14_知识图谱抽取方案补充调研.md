# 知识图谱抽取方案补充调研

> 日期：2026-04-14  
> 目的：为“动力装备知识库”下一阶段知识图谱抽取与 GraphRAG 验证提供可执行候选路线  
> 结论导向：优先选择能在本周内完成 PoC 对比、且与现有 JSON -> chunk -> Chroma 流程衔接最顺的方案

---

## 1. 当前约束

结合目前代码和数据形态，当前知识图谱方案需要满足几个现实条件：

- 输入主要来自 Label Studio 导出的标注 JSON，文本质量不完全整齐，存在 OCR/版面噪声。
- 当前主线已经是“多 JSON 合并 -> 分块 -> 向量化 -> Chroma”，所以知识图谱抽取最好能直接复用现有 chunk。
- 周二前更适合做“小样本、强可比”的 PoC，而不是直接上完整重系统。
- 文档语言以中文专业文本为主，因此中文支持和 schema 可控性比英文 benchmark 成绩更重要。

---

## 2. 候选路线

### 方案 A：LLM Prompt + Schema 约束抽取

核心思路：

- 直接基于现有 chunk，让 LLM 按固定 schema 输出实体、关系、属性三元组。
- 输出 JSON，再做去重、规范化、落图库。

优点：

- 上手最快，和当前 Chroma 流程耦合成本最低。
- 不需要单独准备训练集，适合先做 PoC。
- 方便针对动力装备场景不断改 prompt 和 schema。

缺点：

- 稳定性依赖模型与 prompt 设计。
- 容易出现实体命名不一致、关系漂移、漏抽取。
- 批量处理的 API 成本和质量波动要重点关注。

适合定位：

- 最快验证“知识图谱是否对当前问答有增益”的首选 baseline。

### 方案 B：UIE / DeepKE 这类 schema 驱动的信息抽取工具

核心思路：

- 使用统一信息抽取或知识抽取工具，把实体、关系、属性抽取做成可配置 schema 任务。
- 对中文、低资源、关系抽取、属性抽取更友好。

优点：

- 比纯 prompt 更结构化，schema 可控。
- 适合逐步沉淀领域标签体系。
- 后续可以从 zero-shot 走向 small fine-tuning。

缺点：

- 环境和训练链路比 prompt 更重。
- 如果没有适配好的中文模型或少量标注数据，首轮收益未必立刻超过 prompt。

适合定位：

- 作为“中期可持续路线”的主候选，尤其适合你们这种专业领域知识库。

### 方案 C：REBEL / mREBEL 端到端三元组生成

核心思路：

- 把关系抽取直接建模成 seq2seq 生成任务，从文本端到端输出三元组。

优点：

- 思路直接，天然适合三元组抽取。
- 适合作为“端到端抽取模型”代表方案做对比。

缺点：

- 原始 REBEL 偏英文。
- mREBEL 虽支持多语言，但对专业中文术语和长文档仍需实测。
- 对细粒度工业 schema 的可控性通常不如 prompt + schema 或 DeepKE/UIE。

适合定位：

- 作为研究对照组保留，但不建议作为本周主攻实现。

### 方案 D：Microsoft GraphRAG

核心思路：

- 通过 LLM 从语料中提取结构化图信息，再进行社区发现、层级摘要和局部/全局混合检索。

优点：

- 对“全局主题”“跨段汇总”“多跳关联”问题很强。
- 方法体系完整，适合作为学术与工程上的强参照。

缺点：

- 索引构建成本高，流程比普通 RAG 重得多。
- 更适合“全局问答/总结型问题”，不一定最适合你们当前这种先做精准工程知识查询的阶段。
- 官方仓库也明确提醒 indexing 成本较高。

适合定位：

- 作为后续中长期对照框架，不建议本周直接作为唯一主线。

### 方案 E：LightRAG

核心思路：

- 用更轻量的图结构与检索流程实现 GraphRAG，强调简单、快速、增量更新和工程可用性。

优点：

- 比 Microsoft GraphRAG 更轻，和增量更新场景更契合。
- 官方已提供 server / API / WebUI / Neo4j 等工程入口，落地速度较快。
- 很适合做“图 + 向量”混合检索实验。

缺点：

- 生态活跃，但版本变化快，需要注意接口变动和工程稳定性。
- 如果直接使用其文件上传服务，要特别注意已公开的上传路径穿越漏洞，评估时至少应使用修复版本。

适合定位：

- 如果你们决定做“轻量 GraphRAG 工程验证”，它是比 Microsoft GraphRAG 更现实的优先项。

### 方案 F：KAG / OpenSPG

核心思路：

- 强调逻辑形式、知识对齐、图谱与原始 chunk 双向索引，以及面向垂域知识问答的混合推理。

优点：

- 对规则、时间、数值、专业知识逻辑更重视。
- 如果后续要进入“专业问答 + 规则推理 + 结构化知识服务”，这个方向很值得关注。

缺点：

- 系统较重，不适合作为本周快速验证主线。
- 落地成本和学习成本都高于当前阶段所需。

适合定位：

- 作为后续增强路线保留，不建议本周直接投入主力实现。

## 3. 面向本项目的判断

### 最适合本周做的，不是“谁最先进”，而是“谁最容易与现有流程做出可比结果”

从周二前的节奏看，我更建议把对比收敛到以下三条：

1. `Prompt + Schema`  
   作为最快 baseline，直接复用现有 chunk，最快看到实体/关系抽取结果。

2. `DeepKE / UIE`  
   作为结构化中文抽取路线，代表“更可控、更可持续”的方案。

3. `LightRAG`  
   作为轻量 GraphRAG 路线，代表“图谱增强检索”的工程实现方向。

不建议本周主攻：

- `Microsoft GraphRAG`：研究价值高，但 indexing 成本和工程复杂度偏高。
- `KAG`：很强，但过重。
- `REBEL/mREBEL`：可做补充实验，不建议占主线时间。

---

## 4. 推荐实施顺序

### 第一优先：Prompt + Schema 抽取 PoC

建议直接在现有 chunk 上定义最小 schema：

- 实体类型：`Equipment`、`Component`、`Parameter`、`FaultMode`、`Procedure`、`Standard`
- 关系类型：`HAS_COMPONENT`、`HAS_PARAMETER`、`CAUSES`、`REQUIRES`、`APPLIES_TO`、`THRESHOLD`

首轮目标：

- 只选 20~30 个 chunk 做人工审查。
- 重点看“实体命名一致性”和“关系是否可落图”。

### 第二优先：DeepKE / UIE 小样本验证

建议目标不是一上来训大模型，而是先回答两个问题：

- 在中文动力装备文本里，schema 驱动抽取是否比 prompt 更稳定？
- 在少量人工校正规则下，是否更容易做实体标准化？

### 第三优先：LightRAG 小规模接入验证

建议只做小数据子集：

- 选择一篇或几页典型文档。
- 比较纯向量检索 vs 轻量图增强检索。
- 重点看多跳问答、跨段整合问题是否明显改善。

---

## 5. 周二前建议交付物

为了避免“讨论很多、结论很空”，建议直接产出四个对比件：

1. 一份统一 schema 文档  
   明确实体类型、关系类型、命名规范、例子。

2. 一组共享测试问题  
   至少覆盖：
   - 精确属性查询
   - 设备-部件关系查询
   - 故障原因/处理方法查询
   - 多段信息综合查询

3. 三份小实验结果
   - Prompt + Schema
   - DeepKE / UIE
   - LightRAG 或图增强检索

4. 一页结论表
   维度建议：
   - 开发成本
   - 中文适配
   - 抽取稳定性
   - 检索增益
   - 是否适合继续投入

---

## 6. 我的推荐结论

如果目标是“本周内做出可信的路线判断”，建议：

- 主线落在 `Prompt + Schema` 和 `DeepKE / UIE`
- 轻量补充一个 `LightRAG` 小样本验证
- 暂不把 `Microsoft GraphRAG` 和 `KAG` 作为本周主攻

更具体地说：

- `Prompt + Schema` 负责最快形成结果
- `DeepKE / UIE` 负责验证中文结构化抽取的长期可行性
- `LightRAG` 负责验证“图增强检索”到底值不值得继续

这三条线组合起来，既能回答“知识图谱该不该做”，也能回答“该从哪种工程路线开始做”。
