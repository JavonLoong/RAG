# 清华燃气轮机知识库顶尖对标与验收标准

这份标准用于校准当前 RAG 系统的目标：服务对象是清华大学能源与燃气轮机相关团队，所以不能只按普通开源 RAG demo 的标准验收。结论先写清楚：开源头部不等于行业顶尖。RAGFlow、Dify、Haystack、LlamaIndex、LangChain、Microsoft GraphRAG 这类项目适合作为可复现的工程底座和设计参考，但真正的顶尖还要同时对标闭源企业搜索、高风险文档推理平台、云厂商托管 RAG、科研评测和领域专家验收。

## 对标分层

| 层级 | 代表对象 | 应该学习的东西 | 对本项目的意义 |
| --- | --- | --- | --- |
| 开源成熟工程 | RAGFlow、Dify、Haystack、LlamaIndex、LangChain、Microsoft GraphRAG | 文档解析、索引编排、工具链、GraphRAG local/global search、可本地化部署 | 这是工程下限，不是顶尖终点 |
| 企业级知识搜索 | Glean | 权限继承、连接器、实时同步、企业搜索和 AI assistant | 清华项目也需要权限、审计、可追溯检索，而不是单用户玩具系统 |
| 高风险文档推理 | Hebbia | 多文档、多步骤、带引用的研究工作流和机构级文档分析 | 燃气轮机资料不是闲聊语料，要能做可验证的技术分析 |
| 运营知识系统 | Palantir AIP | Ontology、Document context、Function-backed context、把知识和行动/流程绑定 | 燃气轮机知识库最终应支持部件、故障、工况、实验、维护动作之间的结构化推理 |
| 云厂商托管 RAG | Azure AI Search、Gemini Enterprise Agent Platform、Amazon Bedrock Knowledge Bases | 托管索引、hybrid/agentic retrieval、引用、数据连接、企业级运维 | 给我们提供生产级可靠性和运维能力的标尺 |
| 幻觉与 grounded generation | Vectara | grounded generation、hallucination evaluation、引用优先 | 清华项目必须“宁可拒答，不可胡答” |

## 清华燃气轮机场景的顶尖定义

这个系统不能只回答“某段资料里有什么”。顶尖标准应当是：面对燃气轮机设计、运行、维护、故障诊断、实验数据、标准规范、论文和项目文档，系统能稳定给出有出处、有页码、有证据、有边界条件的回答；遇到证据不足时明确拒答；回答复杂问题时能把文档片段、表格、图谱关系和领域术语组织成可复核的推理链。

## 必须达到的能力

### 文档解析

1. 支持 PDF、扫描 PDF、Word、PPT、Excel、CSV、Markdown、TXT、图片型资料。
2. 支持 layout-aware OCR，保留页码、标题层级、表格、图片标题、公式上下文和图表说明。
3. 对燃气轮机资料要重点保留部件名、型号、工况参数、单位、温度/压力/转速/效率等数值字段。
4. 每个 chunk 必须能回溯到 source_file、page、section、table/figure id 或 OCR 坐标。
5. 文档解析结果必须有质量报告：空页、短块、OCR 低置信度、表格解析失败、页码缺失都要可见。

### 检索质量

1. 默认不是纯向量检索，而是 hybrid retrieval：BM25/keyword + vector + metadata filter + reranker。
2. 支持 query rewrite、术语扩展、中文/英文术语映射，例如 compressor/压气机、combustor/燃烧室、turbine blade/涡轮叶片。
3. 支持精确过滤：文档类型、年份、作者/课题组、设备型号、部件、实验工况、来源目录。
4. 每个回答必须返回引用，引用要到 chunk/page/table/figure 级别。
5. 必须有 no-answer gate：证据不足时拒答，而不是生成看似合理的工程结论。

### GraphRAG

1. 构建燃气轮机领域图谱：部件、子系统、故障模式、传感器、工况、材料、维护动作、实验现象、论文结论。
2. 支持 local search：围绕单个部件或故障做多跳证据检索。
3. 支持 global search：跨文档总结某类主题，例如“燃烧不稳定相关因素”“叶片冷却研究路线”。
4. 社区摘要必须保留 source evidence，不能只有 LLM 总结。
5. 图谱边要有证据文本、source_file、page 和置信度。

### 领域知识

1. 建立燃气轮机术语表和同义词表。
2. 建立单位归一化和数值抽取规则，至少覆盖温度、压力、流量、转速、效率、振动、排放、功率。
3. 建立典型问题模板：故障诊断、设计参数查询、实验结论对比、标准条款定位、部件关系追踪。
4. 关键知识必须经过领域专家抽样审核，而不是只靠通用 RAG 指标。

### 权限与审计

1. 支持 OIDC/JWKS 登录和 HttpOnly session。
2. 支持 role_registry、admin/owner/reviewer/approver 权限。
3. 支持策略变更 proposal/approve/reject/rollback audit。
4. 支持 session inventory、session revoke、session key rotation、session key status。
5. 任何 refresh token、client secret、AESGCM ciphertext、key material 都不能出现在前端或 audit 里。

### 工程可靠性

1. 支持增量 ingestion、失败重试、批处理状态、处理日志。
2. 支持索引版本、备份、回滚、数据目录隔离。
3. 支持 smoke evaluation、regression evaluation、GraphRAG triage regression。
4. 支持离线部署和可复现环境；必要时可对接云厂商托管能力。
5. 所有关键接口必须有测试：解析、检索、GraphRAG、评测、权限、审计、密钥状态。

### 评测门禁

至少建立三套评测集：

1. 通用检索集：单跳事实问答、页码定位、表格查找。
2. 工程推理集：多文档、多跳、参数对比、故障原因链。
3. 安全集：无证据问题、越权资料、提示注入、过时资料、冲突资料。

建议初始验收指标：

| 指标 | 目标 |
| --- | --- |
| citation coverage | 关键回答 100% 有引用 |
| source page accuracy | 专家抽样 >= 95% |
| no-answer precision | 无证据问题 >= 90% 正确拒答 |
| recall@10 | 专家 gold set >= 90% |
| reranked top-3 evidence hit | 专家 gold set >= 85% |
| hallucination rate | 高风险问题 <= 2% |
| permission leak | 0 |
| secret leak | 0 |
| smoke gate | 每次发布必须 pass |

## 当前项目状态判断

当前系统已经不是普通 demo。它已经补了文档解析入口、hybrid retrieval、GraphRAG local/global、评测 smoke、triage regression、OIDC、session、审计、session revoke、session key rotation、session key status 等能力。

但按清华燃气轮机知识库的顶尖要求，还没有到 100%。当前更接近“强工程原型 + 一部分企业治理能力”，不是“可交付给顶尖高校团队长期生产使用的最终系统”。主要差距：

1. 真实燃气轮机语料还没有形成专家 gold set。
2. 文档解析还需要针对扫描 PDF、表格、公式、图表和工程图做真实样本校准。
3. 领域 ontology 还没有完整建立。
4. 术语表、单位归一化、部件/故障/工况抽取规则还不够。
5. 评测指标还没有由清华专家确认阈值。
6. 云 KMS/Secret Manager、自动轮换、备份恢复和运维 runbook 还需要补。

## 后续工作顺序

1. 先拿真实燃气轮机资料做文档解析校准，不再只用通用样例。
2. 建立燃气轮机术语表、同义词表、单位规则和领域 ontology。
3. 建立专家 gold set 和验收指标。
4. 把 retrieval / GraphRAG / no-answer gate 的默认策略按 gold set 自动调参。
5. 补云 KMS 或本地 HSM/Secret Manager 抽象、自动轮换和运维 runbook。
6. 做面向清华团队的最终演示工作流：上传资料、解析质量检查、问答、证据追踪、专家反馈、回归评测。
