# 30题评测集说明

## 目标

这份评测集用于把当前 RAG 项目从“能演示”推进到“能评估”。它不追求题量很大，而是覆盖课程汇报中最需要证明的几类能力：

- 普通 RAG 能不能从动力装备资料中找回事实证据。
- 检索结果能不能支撑带依据的回答。
- OCR 和 JSON 入库风险是否被识别和控制。
- 知识图谱 POC 是否能用 evidence 和人工评审证明链路走通。
- 评测指标是否能支撑后续对比实验。

机器可读数据集位于：

`evaluation/system_eval_questions.jsonl`

## 题目分布

| 类别 | 题号 | 数量 | 主要验证点 |
| --- | --- | ---: | --- |
| 普通 RAG 事实题 | se001-se006 | 6 | 燃气轮机、联合循环、核心部件等基础知识召回 |
| 流程与工程题 | se007-se010, se013 | 5 | 启动、监测、维修报告、reranker 等流程理解 |
| 对比与回答质量 | se011-se015 | 5 | 向量检索、关键词检索、Hybrid、citation、防幻觉 |
| OCR / JSON 风险 | se016-se020 | 5 | OCR 可靠性、版面噪声、Label Studio JSON 入库 |
| KG / GraphRAG | se021-se026 | 6 | schema、三元组、evidence、局部/全局搜索、社区摘要 |
| 结构化数据与评测方法 | se027-se030 | 4 | Goldwind 数据质量、context recall、成功/失败案例分析 |

## 字段说明

每行是一条 JSON 记录，兼容 `scripts/run_system_evaluation.py`：

- `id`：题目编号。
- `question`：评测问题。
- `reference_answer`：人工标准答案，用于汇报和后续人工评分。
- `expected_evidence_keywords`：检索证据中应覆盖的关键词，用于自动计算关键词召回。
- `task_type`：题目类型，用于分组统计。
- `source_scope`：预期资料来源范围。
- `grading_notes`：人工评分说明。
- `difficulty`：题目难度。

## 汇报口径

可以这样讲：

> 为了避免只做 demo，我补了一套 30 题小型评测集。题目覆盖事实问答、流程理解、检索策略对比、OCR 风险、知识图谱 POC 和评测方法。每道题都有标准答案和预期证据关键词，后续可以用统一脚本统计 retrieval recall、answer evidence coverage、citation 缺失率和 hallucination risk。

## 下一步

第 3 天建议基于这 30 题分别跑：

1. 向量检索 baseline。
2. BM25 / 关键词检索 baseline。
3. Hybrid Search。
4. 如果当前链路稳定，再补 Hybrid + Reranker。

最终汇报时至少形成一张对比表，并选 2 个成功案例、2 个失败案例做解释。
