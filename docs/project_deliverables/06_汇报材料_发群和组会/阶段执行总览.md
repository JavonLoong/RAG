# RAG / GraphRAG 阶段成果汇总 2026-05-15

这次不是做展示页，而是把老师给的真实资料接进工程链路，验证“资料能不能处理、能不能入库、能不能检索、能不能抽关系、问题能不能被记录”。

## 1. PDF 可抽取性判断

产出：

- `scripts/audit_pdf_text_extractability.py`
- `data_pipeline/reports/pdf_extractability_report.md`
- `data_pipeline/reports/pdf_extractability_report.json`
- `data_pipeline/reports/pdf_extractability_report.csv`

结果：

- 共检查 14 本 PDF，合计 6098 页。
- 只有 1 本可直接抽文本。
- 13 本需要 OCR。
- 唯一可直接抽文本的是《燃气-蒸汽联合循环发电机组运行技术问答...》。

这说明老师给的资料不能直接全部入 ChromaDB，必须先分出“可抽文本”和“OCR 队列”。

## 2. 普通 RAG 的 ChromaDB

产出：

- `scripts/build_representative_rag_chroma.py`
- `storage_layer/runtime/representative_rag_chroma/`
- `docs/representative_rag_chroma_build.md`

结果：

- Collection：`gas_turbine_representative_rag`
- 入库 593 条 chunk。
- 其中 480 条来自 QA JSONL，113 条来自可抽文本 PDF。
- 已验证 ChromaDB 持久化库可重新打开，`collection.count()` 为 593。

边界：

- 当前使用本地 hashing embedding，只证明链路跑通，不代表最终检索效果。

## 5. GitHub Pages 页面怎么复用

产出：

- `docs/github_pages_rag_console_integration.md`
- `frontend_app/current_console/README_integration.md`

结论：

- `https://javonloong.github.io/RAG/` 可以作为控制台前端复用。
- 但 GitHub Pages 是静态页面，不能自己跑 Python、ChromaDB、OCR、文件入库。
- 正确结构是：GitHub Pages 前端 -> 后端 API -> ChromaDB -> 返回检索结果。
