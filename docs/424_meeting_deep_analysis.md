## 会议实际分成三条主线

### 1. 当前系统工程问题：日志、超时、大文件处理

会议一开始讨论的是现有工具在处理文件时的问题：

- 上传或处理速度慢，可能和 Q 资源、文件体积、CPU 占用有关。
- 某些文件处理时 CPU 占用很高，可能达到 100%。
- 部分文件无法处理，前端或操作日志只显示简单错误，后端没有足够详细的错误信息。
- 大文件可能达到 60MB、数百万 token，处理时间长是正常风险，但现在无法判断是“文件太大导致超时”，还是“文件内容/格式不规范导致程序报错后表现为超时”。

老师的明确建议是：不要只在前端显示“成功/失败”，需要把后端处理过程写入 `.log` 文件，方便定位问题。前端可以只展示简洁状态，例如处理成功、处理失败、入库数量等；详细日志放到文件里。

这说明你们当前系统的首要工程改进点是“可观测性”，也就是出问题时能知道错在哪里。

### 2. 知识图谱方向：先做 Graph，再谈 GraphRAG

你在会上提到自己之前误以为知识图谱主要是一种可视化形式，后来意识到它本质上是数据结构：实体、关系和属性组成的图。

老师基本认可这个理解，并进一步强调：

- Neo4j 本质上是图数据库，是知识图谱常见的存储方式。
- Microsoft GraphRAG 是完整方法路线：先从文本建图，再基于图做检索增强生成。
- LlamaIndex / LangChain 更像快速接入现有 RAG 项目的框架工具。
- 但你们当前最应该明确的是：现在到底是在做 graph，还是在做 RAG。

会议里的关键判断是：你们目前更应该先聚焦“从文档中抽取实体关系并构建 graph”，GraphRAG 里用于问答的 RAG 部分可以放在后面。换句话说，当前阶段的主要任务不是马上做一个完整 GraphRAG 问答系统，而是先证明“能不能从燃气轮机领域文档里抽出靠谱的实体关系图”。

## 关键概念校准

### 知识图谱不是图片

知识图谱本质是数据结构，不是可视化图片。图片只是展示方式。

可以这样理解：

```text
实体：燃气轮机、压气机、齿轮箱、润滑油
关系：包含、连接、导致、属于、处理建议
属性：型号、时间、来源文档、置信度
```

真正有用的是系统能沿着这些关系查询和推理，而不是图画得多漂亮。

### Graph 和 GraphRAG 的关系

Graph 是知识结构，GraphRAG 是用这个结构增强问答。

更准确地说：

```text
第一步：从文档中抽取实体关系，构建知识图谱
第二步：用图谱帮助检索、组织上下文
第三步：把检索到的图谱关系和原文证据交给大模型生成答案
```

会议里反复讨论的偏差就在这里：你们目前更缺的是第一步，也就是 graph construction。

### Neo4j、Microsoft GraphRAG、LlamaIndex / LangChain 的区别

| 方向 | 它是什么 | 当前应该怎么用 |
| --- | --- | --- |
| Microsoft GraphRAG | 完整 GraphRAG 方法和工具链 | 用来理解标准流程：抽实体、抽关系、社区摘要、全局/局部检索 |
| Neo4j | 图数据库 | 用来存储和查询知识图谱，不等于自动完成抽取 |
| LlamaIndex / LangChain | RAG/LLM 应用框架 | 用来快速做三元组抽取、图索引、检索集成 |

所以三者不是同一类东西，不能简单横向说“哪个好”。更合理的表达是：

```text
Microsoft GraphRAG 看方法论；
Neo4j 看图谱落库；
LlamaIndex / LangChain 看快速集成和实验。
```

## 文档处理/切分问题值得单独注意

会议里还提到一个很重要的问题：双栏文档或复杂版式文档，如果按从上到下顺序拼接，可能会把左栏和右栏文本错误拼在一起，导致段落语义错乱。

这会直接影响 RAG 和知识图谱抽取，因为实体关系抽取依赖上下文。如果前面的文本顺序已经乱了，后面抽出来的关系也会乱。

建议后续把文档处理列成一个技术风险：

- 对双栏文档，需要根据坐标或语义判断阅读顺序。
- chunk 不宜太短，否则上下文断裂，关系抽取不稳定。
- 可以考虑父子 chunk：大 chunk 保留完整语义，小 chunk 用于精细检索。
- OCR 或版面解析结果要保留坐标，方便后续重组段落。
