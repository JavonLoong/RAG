# 面向清华挑战杯特等奖的动力装备可信 GraphRAG 项目升级设计

生成日期：2026-06-05

## 1. 目标与定位

本项目从“RAG 课程结项工程”升级为“面向动力装备运维知识的可信 GraphRAG 系统”。参赛叙事采用“能源/动力装备应用问题牵引，信息技术方法支撑”的双线定位：

- 应用问题：动力装备资料分散、扫描件多、术语密集，部件、故障、参数、工况和处理措施之间的证据链难以被普通检索系统稳定追溯。
- 技术方法：把 OCR、结构化知识抽取、普通 RAG、混合检索、GraphRAG、证据绑定、幻觉风险诊断和自动评测组织成可复现工程闭环。
- 竞赛成果：形成一个能现场演示、能结项提交、能用实验数据支撑创新性的项目包，而不是只展示一个问答页面。

建议参赛名称：

> 知燃知维：面向动力装备运维知识的可信 GraphRAG 系统

## 2. 当前事实基线

以下事实来自当前工作树中的 README、交付材料、评测材料和测试结果，作为升级设计的起点：

- 普通 RAG 数据库已构建，包含 9080 个 chunk，其中 OCR 文本 8468 个、直接抽文本 PDF 132 个、问答 JSON 480 个。
- 当前系统评测集包含 30 题，覆盖普通 RAG、OCR 风险、GraphRAG/知识图谱、结构化数据和评测方法。
- Graph construction POC 已完成人工评审闭环：27 条候选三元组中 26 条正确、1 条待讨论、0 条明确错误。
- 控制台后端具备上传、处理、入库、统计、搜索、导出、日志和 GraphRAG 相关 API。
- 项目已存在课程汇报最终交付包，包含 PPT、讲稿、评测说明、失败案例分析、演示脚本、备用证据包和答辩口径。
- 当前测试基线已验证：`tests/unit` 通过 74 项，`api_server/current_console/chroma_rag_poc/tests` 通过 19 项。

## 3. 评审级差距

若按清华挑战杯特等奖候选项目标准审视，当前项目的主要差距不是“没有功能”，而是证据链还没有被组织成竞赛作品：

1. 项目主张仍偏课程汇报，缺少挑战杯项目书级别的问题定义、创新点、应用价值和成果摘要。
2. 评测集只有 30 题，能够支撑课程结项，但不足以证明技术路线在动力装备知识任务上的稳定收益。
3. 现有 baseline 主要比较 keyword、hashing dense 和 hybrid RRF，缺少真实 embedding、reranker 和 GraphRAG 同题对比。
4. GraphRAG POC 已证明 evidence-bound graph construction，但还没有把普通 RAG、GraphRAG local、GraphRAG global 的增益放到同一个评测框架中展示。
5. 演示包偏“课程最终包”，还缺少挑战杯评审入口：一页式成果总览、项目书、技术白皮书、实验报告、现场答辩脚本和可复现实验说明。
6. 当前工作树里有大量未提交删除和迁移痕迹，容易让评审入口显得松散，需要建立一个清晰的 `challenge_cup` 成果目录承接最终材料。

## 4. 推荐路线

推荐采用“动力装备可信知识工程路线”，而不是纯课程结项路线或纯信息技术路线。

### 4.1 不采用纯课程结项路线

课程路线只需要材料齐全、演示可跑、测试通过。它能满足结项，但无法解释为什么能进入挑战杯特等奖答辩。

### 4.2 不采用纯信息技术路线

纯信息技术路线会强调 GraphRAG、检索、重排和幻觉守卫，但容易被评委追问真实应用价值、行业数据来源和落地场景。当前项目最大的优势是动力装备资料和运维知识，不应削弱这个优势。

### 4.3 采用动力装备可信知识工程路线

最终叙事聚焦于：

- 真实资料链路：从扫描 PDF、OCR、清洗、chunk 到 ChromaDB。
- 可信知识链路：从文本证据到实体、关系、三元组、人工评审和图谱可视化。
- 可评测检索链路：同一批问题下比较普通 RAG、混合检索和 GraphRAG。
- 可解释回答链路：回答必须返回证据片段、图谱关系、引用和不足说明。
- 可复现实验链路：评测集、脚本、报告、失败归因和演示包全部落地到仓库。

## 5. 系统设计

### 5.1 目标用户

第一目标用户是挑战杯评委和结项评审老师。他们需要在短时间内判断项目是否有真实问题、技术创新、完成度和可验证成果。

第二目标用户是动力装备课程、运维知识整理和燃气轮机资料学习场景中的研究者或学生。他们需要在专业资料中追溯定义、参数、故障原因、部件关系和处理建议的证据来源。

第三目标用户是后续开发者。他们需要可以复跑评测、替换 embedding/reranker、扩展 GraphRAG schema，并知道每个实验结果来自哪个脚本和数据版本。

### 5.2 竞赛级能力闭环

系统最终需要支持以下闭环：

1. 数据输入：接入 OCR 文本、直接抽文本 PDF、问答 JSON 和知识图谱 POC 输出。
2. 质量审计：记录 OCR 风险页、低置信度页、文本清洗结果和可入库 chunk 统计。
3. 普通 RAG：支持 keyword、dense、hybrid 和 rerank 检索模式，返回证据、来源和分数。
4. GraphRAG：支持 evidence-bound triples、图谱统计、社区检测、社区摘要、局部图检索和全局摘要检索。
5. 答案生成：回答时区分证据充分、证据不足和需要人工判断的情况，并保留 citation。
6. 幻觉诊断：对答案中的数值、实体、因果和处理建议做 evidence support 检查。
7. 自动评测：同题比较不同检索/问答策略，输出 recall、关键词覆盖率、证据覆盖率、citation 缺失率、幻觉风险和耗时。
8. 现场演示：用固定样例展示资料入库、证据检索、图谱关系、回答引用、失败归因和改进前后对比。

### 5.4 工程模块设计

工程改造保持现有模块边界，不做大规模搬家：

- `evaluation/`：扩展评测集、指标报告和同题对比输出。
- `scripts/`：新增挑战杯报告生成、评测汇总和成果打包脚本。
- `rag_orchestrator/`：补强回答证据、全局搜索和幻觉诊断的评测接口。
- `retrieval_engine/`：把 hybrid/reranker 能力对齐到评测脚本和 API 搜索模式。
- `api_server/current_console/`：保持可运行入口稳定，补充 GraphRAG/hybrid 搜索演示路径，不在第一阶段拆分大型路由。
- `frontend_app/current_console/`：优先做演示状态和证据展示，不重写前端框架。
- `docs/challenge_cup/`：承接项目书、白皮书、实验报告和答辩材料。

### 5.5 数据流

```text
原始资料 / OCR 文本 / JSON
  -> 数据清洗与 chunk
  -> 普通 RAG 索引
  -> 检索 baseline
  -> Graph construction
  -> evidence-bound triples
  -> 社区检测与摘要
  -> RAG / Hybrid / GraphRAG 同题回答
  -> 自动评测与失败归因
  -> 挑战杯材料与现场演示
```

### 5.6 错误处理与边界口径

项目材料中必须明确以下边界，避免过度声称：

- 可以说：本项目已形成动力装备资料处理、普通 RAG、Graph construction POC、评测和演示闭环。
- 可以说：GraphRAG 的关键增量是 evidence-bound relationships、局部关系检索和社区摘要，全局检索用于跨文档归纳类问题。
- 可以说：当前系统支持评测驱动迭代，可以比较不同检索策略和失败原因。
- 不说：系统已经替代工程师完成真实运维决策。
- 不说：所有三元组均由 LLM 自动抽取并在大规模真实生产库中完全验证。
- 不说：GraphRAG 在所有问题上必然优于普通 RAG。
- 不说：当前 30 题评测集已经足以作为最终论文级 benchmark。

## 6. 实验设计

### 6.1 评测集扩展

评测集从 30 题扩展到 60 题。新增题目必须覆盖：

- 定义类：燃气轮机、燃烧室、压气机、联合循环等核心概念。
- 部件关系类：设备、部件、参数、工况之间的显式关系。
- 故障诊断类：现象、可能原因、检查项、处理措施和证据来源。
- OCR 风险类：扫描件、两栏排版、低置信度页对 RAG 的影响。
- 结构化事实类：项目统计、评测指标、图谱三元组数量和人工评审结论。
- 全局归纳类：跨资料总结问题，用于检验 GraphRAG global 的价值。

每题需要包含：

- `id`
- `question`
- `reference_answer`
- `expected_evidence_keywords`
- `task_type`
- `source_scope`
- `expected_modes`
- `grading_notes`

### 6.2 方法对比

第一阶段必须跑通以下方法：

| 方法 | 目的 | 说明 |
| --- | --- | --- |
| keyword | 词法基线 | 当前已能跑，是稳健下限 |
| dense_hashing | 流程基线 | 证明 dense 流程，但不作为最终质量代表 |
| hybrid_rrf | 混合召回 | 当前已有结果，继续保留 |
| hybrid_rerank | 检索增强 | 使用可配置 reranker 或离线重排替代方案 |
| graphrag_context | 图谱证据上下文 | 对比普通 RAG 证据覆盖 |
| graphrag_global | 社区摘要/全局归纳 | 专门服务跨文档总结问题 |

第二阶段在环境允许时加入真实 embedding：

- 中文/多语技术文档 embedding，如 BGE-M3 或同等级模型。
- 记录模型名称、版本、运行环境、耗时和失败回退策略。

### 6.3 指标

主指标：

- question recall@K
- keyword recall@K
- evidence coverage
- citation coverage
- hallucination risk
- answer completeness

辅助指标：

- 平均检索耗时
- 平均上下文长度
- 失败原因类别分布
- GraphRAG 构建成本
- 人工评审通过率

### 6.4 成功门槛

结项目标门槛：

- 60 题评测集可被脚本读取并生成报告。
- 至少 4 种检索/问答策略有同题对比结果。
- 项目书、技术白皮书、实验报告、演示脚本和结项验收清单齐全。
- 本地控制台和离线备用证据包均可用于答辩。

挑战杯冲奖门槛：

- hybrid 或 GraphRAG 策略在目标问题子集上相对 keyword baseline 有可量化提升。
- GraphRAG global 在全局归纳类问题上展示普通 RAG 不易覆盖的证据组织优势。
- 每个核心结论都有材料、脚本或报告作为证据。
- 答辩材料能清晰说明“不替代工程师决策，只提供可追溯证据型辅助”的安全边界。

## 8. 验收标准

### 8.1 工程验收

- 根目录单元测试通过。
- 控制台后端测试通过。
- 评测脚本能读取扩展后的评测集并输出报告。
- 成果包脚本能生成 `docs/challenge_cup` 下的核心材料清单。
- 关键演示路径在本地可运行或有离线备用证据。

### 8.2 材料验收

- 项目书包含问题背景、相关工作、技术路线、创新点、实验结果、应用价值、团队分工和风险边界。
- 技术白皮书能让技术评委理解系统架构、数据流、GraphRAG 增量、评测方法和工程实现。
- 实验报告包含方法对比、指标、失败案例和结论，不只展示成功样例。
- 答辩问答手册覆盖“为什么需要 GraphRAG”“是否真实可用”“是否会幻觉”“数据是否合法”“为什么能冲特等奖”等问题。
- 结项验收清单能逐项映射到文件、脚本、报告和演示证据。

### 8.3 竞赛验收

每个核心主张必须对应证据：

| 主张 | 证据 |
| --- | --- |
| 资料处理链路完整 | OCR 审计报告、数据库构建报告、chunk 统计 |
| 知识图谱不是空图 | 三元组表、evidence 页面、人工评审总结 |
| RAG 能被评测 | 评测集、脚本、baseline 报告 |
| GraphRAG 有增量价值 | GraphRAG 同题报告、全局归纳案例 |
| 系统可演示 | 控制台、演示脚本、备用证据包 |
| 项目边界严谨 | 答辩手册、技术白皮书、安全边界章节 |

## 9. 实施分解

后续实施拆成四条工作流：

1. 材料工作流：创建 `docs/challenge_cup/`，生成项目书、白皮书、实验报告、答辩手册和结项清单。
2. 评测工作流：扩展评测集，新增多策略对比报告，补充 GraphRAG 子集指标。
3. 工程工作流：把 hybrid/reranker/GraphRAG 搜索能力对齐到脚本和 API 演示路径，保持现有控制台稳定。
4. 演示工作流：生成固定演示问题、截图、备用证据包和最终打包脚本。

第一轮实施优先级：

1. 先建 `docs/challenge_cup/` 高层入口和验收清单。
2. 扩展评测集到 60 题并生成静态评测说明。
3. 生成挑战杯项目书和技术白皮书初版。
4. 跑当前可用评测脚本，复用已有 baseline 报告形成竞赛级实验报告。
5. 再决定是否接入真实 embedding/reranker 的重实验。

## 10. 不纳入第一轮的内容

以下内容不作为第一轮完成标准：

- 重写前端为新的框架。
- 引入生产级用户系统、权限系统和云部署。
- 宣称真实工业现场闭环决策。
- 大规模 Neo4j 生产部署。
- 训练专用大模型。
- 完整论文投稿级统计显著性实验。

这些内容可以进入挑战杯后续优化路线，但不应阻塞结项和第一版竞赛包。
