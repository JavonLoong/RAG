# PowerRAG 项目介绍与复盘报告

> 版本日期：2026-06-21  
> 适用场景：项目汇报、开源 README 扩展说明、阶段复盘、答辩前项目梳理、后续迭代任务对齐。

## 1. 一句话介绍

PowerRAG 是一个本地优先的 RAG / GraphRAG 知识工作台，用于把 PDF、JSON、文本、聊天记录等非结构化资料转化为可检索、可追溯、可评估的知识系统，并通过标准 RAG、混合检索、知识图谱、社区摘要、全局搜索和 Electron 桌面端提供完整的问答与分析能力。

它不是一个单纯的“调用大模型聊天页面”，而是围绕“证据检索、知识结构化、图谱推理、引用可追溯、评估可复现”构建的本地知识系统。

## 2. 项目定位

本项目的核心目标是解决一个具体问题：

> 当资料规模变大、来源复杂、问题需要跨文档或跨联系人聚合时，单纯把文本丢给大模型或只做 top-k 向量检索，会出现召回不全、证据断裂、关系缺失、回答不可审计的问题。

PowerRAG 的定位是：

- 面向本地资料的知识库构建系统。
- 面向复杂问题的 RAG / GraphRAG 实验平台。
- 面向项目汇报和研究验证的可视化控制台。
- 面向真实使用的桌面端本地工具。
- 面向持续迭代的工程化原型，而不是一次性脚本。

当前项目尤其强调三件事：

1. **证据优先**：回答必须能回到原始 chunk、文档、消息、图谱边或社区摘要。
2. **结构补充**：不仅检索文本，也抽取实体、关系、图谱和社区结构。
3. **可复盘**：每一次坏回答都要能解释是 query router、文本召回、图检索、社区摘要、LLM 还是前端状态的问题。

## 3. 项目已经具备的能力

### 3.1 标准 RAG

标准 RAG 部分负责文档导入、文本解析、chunk 切分、本地检索和基于引用证据的问答。

现有能力包括：

- 文件导入与入库。
- JSON / TXT / PDF 等资料解析。
- chunk 切分与 overlap 控制。
- 本地向量或混合检索。
- LLM 问答配置。
- 检索结果展示与 citation 展示。
- 基础压测、统计与处理日志。

这一部分适合回答：

- 某个资料里是否提到某个概念？
- 某个术语、方法、定义是什么？
- 某个文档中的局部事实是什么？
- 需要引用原文证据的普通问答。

### 3.2 GraphRAG

GraphRAG 部分负责把文本资料进一步抽取成实体、关系和图谱结构，并用图结构补充标准 RAG 的不足。

现有能力包括：

- 图谱语料接入。
- Graph Schema 配置。
- 实体类型与关系类型配置。
- 基于规则、Schema 或泛化模式的实体关系抽取。
- 图谱可视化。
- 图谱 JSON 导出。
- 图谱导入后端 GraphStore。
- Leiden 社区检测。
- 社区摘要。
- 全局搜索。
- 局部图谱检索。
- GraphRAG triage 记录与质量分析。

GraphRAG 适合回答：

- 哪些实体之间存在关系？
- 某个实体和哪些主题、人物、事件相连？
- 一个问题需要跨多个 chunk 聚合时，哪些社区摘要可以提供全局线索？
- 为什么一个回答没有召回足够证据？
- 某个复杂关系是否有原文支撑？

### 3.3 混合检索与路由

项目中已经存在多个检索和编排模块：

- `retrieval_engine/hybrid.py`：混合检索。
- `retrieval_engine/graph.py`：图检索。
- `rag_orchestrator/router.py`：查询路由。
- `rag_orchestrator/query_understanding.py`：问题理解。
- `rag_orchestrator/global_search.py`：基于社区摘要的全局搜索。
- `rag_orchestrator/graphrag_qa.py`：GraphRAG 问答编排。

这说明项目不是只有单一 top-k chunk 方案，而是已经具备多路径检索的雏形。

理想流程应当是：

```text
用户问题
  ↓
Query Router 判断问题类型
  ↓
选择检索路径
  ├─ 标准向量 / 关键词 / 混合检索
  ├─ 局部图谱实体邻域检索
  ├─ 全局社区摘要检索
  └─ 特殊任务型扫描，例如全量联系人级分析
  ↓
证据合并与去重
  ↓
LLM 基于证据回答
  ↓
输出 citation / graph evidence / route / capability
```

### 3.4 评估系统

项目中已经有 `evaluation/` 目录和大量报告文件，说明它不只是功能演示，还在尝试建立评估闭环。

现有评估方向包括：

- RAG smoke evaluation。
- GraphRAG triage regression。
- retrieval pipeline 测试。
- query understanding 测试。
- graph store 测试。
- community summary evidence 测试。
- external benchmark 对接。

这类能力的价值是：

- 发现回答质量下降。
- 把失败案例转成回归测试。
- 比较不同检索模式。
- 让系统优化不是靠感觉，而是靠可复现实验。

### 3.5 Electron 桌面端

项目已经加入 Electron 桌面端：

- `electron/main.cjs`
- `electron/preload.cjs`
- `electron/README.md`

桌面端的意义是：

- 用户不必理解浏览器、服务端、端口和本地文件权限。
- 可以通过桌面壳启动本地后端。
- 可以为大规模本地文件处理提供更高内存和更稳定的运行环境。
- 可以通过受限 IPC 做安全的本地文件选择。

这对本项目很关键，因为 RAG / GraphRAG 系统经常需要处理较大的本地文件，浏览器原生环境容易受权限、内存和渲染限制影响。

### 3.6 私有语料分析能力

项目近期加入了微信私聊 RAG 语料导入和 GraphRAG 一键检测流程。

它的工程意义不在于“分析聊天记录”本身，而在于验证一个更通用的问题：

> 当数据天然按联系人、会话、时间、消息方向分区时，RAG 系统必须保留原始层级结构，否则后续图谱构建和问答都会失真。

这类语料对系统提出了更高要求：

- 必须保留 `contact_name`、`username`、`conversation_id`。
- 必须保留 `message_id`、`chunk_id`。
- 必须保留发送方、时间和原文。
- 必须排除群聊、公众号、系统通知、文件传输助手等噪声。
- 不能只基于 top-k chunk 做“全量联系人级”判断。

因此，这个能力可以作为项目的复杂数据场景验证样例。

## 4. 当前项目结构说明

当前主结构可以理解为七层：

```text
RAG/
├─ api_server/                 后端服务与 API 网关
├─ frontend_app/               Web 控制台
├─ electron/                   桌面端壳
├─ data_pipeline/              数据导入、解析、清洗、切分
├─ retrieval_engine/           向量、关键词、图谱、混合检索
├─ rag_orchestrator/           查询路由、问答编排、全局搜索
├─ kg_pipeline/                知识图谱抽取、社区检测、社区摘要
├─ storage_layer/              本地运行数据、图存储、向量存储
├─ evaluation/                 自动化评估与报告
├─ tests/                      单元测试、回归测试、契约测试
├─ docs/                       说明文档、复盘、交付材料
└─ scripts/                    工程脚本
```

各模块职责如下。

| 模块 | 职责 | 项目价值 |
|---|---|---|
| `api_server/current_console` | 当前可运行后端入口 | 保证系统能作为本地服务运行 |
| `frontend_app/current_console` | 控制台页面 | 承载导入、检索、构图、问答、评估入口 |
| `electron` | 桌面端启动器 | 降低使用门槛，改善本地文件处理体验 |
| `retrieval_engine` | 检索能力 | 负责召回证据 |
| `rag_orchestrator` | 编排与路由 | 决定问题走哪条路径 |
| `kg_pipeline` | 图谱能力 | 支撑 GraphRAG 和全局结构分析 |
| `storage_layer` | 数据持久化 | 保存本地索引、图谱、运行状态 |
| `evaluation` | 评估闭环 | 把好坏回答变成可量化问题 |

## 5. 核心流程

### 5.1 标准 RAG 流程

```text
文件上传
  ↓
解析文本 / OCR / JSON 结构化读取
  ↓
切分 chunk
  ↓
写入本地检索集合
  ↓
用户提问
  ↓
检索 top-k 证据
  ↓
LLM 基于证据回答
  ↓
显示 answer + citation
```

优点：

- 快。
- 简单。
- 适合局部事实问答。

风险：

- top-k 容易漏掉分散证据。
- 对全量统计、跨实体关系、全局总结不稳定。
- 如果 chunk 缺少元数据，后续很难追溯。

### 5.2 GraphRAG 流程

```text
语料文件
  ↓
解析为结构化文本记录
  ↓
根据 Graph Schema 抽取实体与关系
  ↓
生成图谱节点和边
  ↓
导入后端 GraphStore
  ↓
社区检测
  ↓
社区摘要
  ↓
局部搜索 / 全局搜索 / 混合问答
```

优点：

- 更适合跨 chunk 的关系问题。
- 能做实体邻域扩展。
- 能通过社区摘要支持全局问题。
- 能把“为什么没答出来”定位到图、文本或摘要缺失。

风险：

- 图谱抽取质量依赖 schema。
- 如果原文证据没有绑定，图谱会变成不可审计的实体词云。
- 图谱过大时前端渲染必须降级，否则会卡死。

### 5.3 桌面端流程

```text
用户启动 Electron
  ↓
Electron 检查后端健康状态
  ↓
必要时启动本地后端
  ↓
打开控制台
  ↓
通过受限 IPC 选择本地文件
  ↓
前端执行导入 / 构图 / 问答
```

桌面端的定位不是替代后端，而是把本地工具变得更像一个可以交付给用户的应用。

## 6. 项目亮点

### 6.1 不是只有向量检索

项目已经具备标准 RAG、混合检索、图检索、社区摘要和全局搜索的路线。这比很多只包装向量数据库的 Demo 更接近真实 RAG 系统。

### 6.2 强调 evidence

项目持续围绕 citation、chunk、message_id、graph edge、community summary 做证据追溯，这是 RAG 系统避免幻觉的关键。

### 6.3 有评估闭环

项目中存在评估报告、triage 记录和回归测试思路。这说明它不是单次演示，而是能把失败案例沉淀下来。

### 6.4 有本地桌面端

Electron 桌面端让项目从“网页原型”向“本地工具”迈进。对私有资料、PDF、聊天记录、个人知识库等场景很重要。

### 6.5 已经暴露并处理真实问题

项目迭代中遇到过：

- 图谱残留导致问答引用旧数据。
- 前端渲染大图卡死。
- top-k chunk 无法回答全量联系人级问题。
- 图谱有结构但问答没有使用到图证据。
- JSON 语料缺少层级结构导致后续分析失真。

这些问题不是坏事，反而说明项目已经进入真实系统会遇到的复杂区，而不是停留在玩具样例。

### 7.1 已完成的关键工作

1. 搭建了 RAG 控制台主线。
2. 拆分出后端、前端、存储、检索、编排、图谱、评估等模块。
3. 增加了 GraphRAG 相关 API。
4. 引入社区检测、社区摘要、全局搜索能力。
5. 加入 GraphRAG triage 和评估机制。
6. 增加 Electron 桌面端。
7. 针对私有聊天记录场景修复了语料结构、消息方向和 chunk 元数据问题。
8. 对前端导入和构图流程加入进度与日志。
9. 对大图渲染做了降级，避免浏览器因渲染过大图谱卡死。
10. 增加了微信私聊一键 GraphRAG 检测入口。

### 7.2 主要问题

#### 问题 1：系统能力有，但前端没有全部接起来

后端已经有 router、global search、community summary、triage 等能力，但用户在前端问答时不一定能感知系统具体走了哪条路径。

后果：

- 用户以为系统只在看图谱。
- 用户不知道为什么检索不到文本证据。
- 用户无法判断是数据没入库、图没构好、路由没走对，还是 LLM 没配。

应对：

- 问答结果必须展示 route、capabilities、text evidence、graph evidence、global evidence。
- 前端需要流程图或执行轨迹。
- 对“没有上下文”的回答要显示具体缺失项。

#### 问题 2：全量问题不能用普通 top-k 思路解决

例如“有几个联系人和我存在暧昧关系”这种问题，不是找 20 个 chunk 就能解决，而是联系人级扫描问题。

正确路径应是：

```text
识别任务类型：全量联系人级分析
  ↓
按联系人分区
  ↓
每个联系人内部召回候选证据
  ↓
逐联系人判断
  ↓
输出统计和未计入原因
```

如果仍然按 top-k chunk 走，必然漏人。

#### 问题 3：图谱不是答案本身

图谱中出现“喜欢”“宝宝”“果果”等实体，不等于可以判断某个联系人暧昧。必须回到原始消息证据。

正确原则：

- 没有原文证据，不允许判断。
- 没有日期、发送方、message_id 或 chunk_id，不应作为强证据。
- 图谱关系只能辅助召回，不能替代证据。

#### 问题 4：前端大图渲染容易误导用户

图谱节点很多时，强行渲染会非常慢，甚至让用户以为系统卡死。

正确策略：

- 大图只显示统计与抽样。
- 完整图谱保留在后端或导出文件。
- 问答和导出继续使用完整数据。
- 前端展示必须和后端数据能力区分开。

#### 问题 5：文档门面不足

项目能力很多，但如果 README、复盘文档、演示流程不清楚，会让人误以为系统杂乱。

应对：

- 保留当前结构。
- 补充项目介绍、架构说明、运行入口、验证命令、已知限制。
- 把缺陷写成“当前边界”和“下一阶段计划”。

## 8. 当前最应该坚持的工程原则

### 8.1 不重构主结构

当前目录虽然复杂，但已经形成模块边界。短期内最重要的是补文档、补验证、补入口，不要继续大规模移动文件。

### 8.2 保留原始证据

所有 RAG 和 GraphRAG 输出最终都要能回到原始文本。

对聊天记录尤其如此：

- 联系人是谁。
- 哪一天。
- 谁发的。
- 原文是什么。
- 属于哪个 chunk。
- 属于哪个 message。

### 8.3 区分四类检索

系统应明确区分：

1. 局部事实检索。
2. 实体关系检索。
3. 全局社区摘要检索。
4. 全量分区扫描任务。

不同问题必须走不同路径。

### 8.4 失败要可解释

一个成熟系统不应该只说“没有上下文”。它应该说：

- 文本检索是否为空。
- 图检索是否为空。
- 社区摘要是否为空。
- router 是否启用。
- LLM 是否配置。
- 当前集合是什么。
- 当前图谱节点和边数量是多少。

### 8.5 前端要展示过程

复杂系统必须让用户看到过程，包括：

- 当前阶段。
- 当前进度。
- 当前处理文件。
- 当前使用的检索路径。
- 当前证据来源。
- 当前失败原因。

## 10. Demo 讲解顺序

### 10.1 标准演示路径

1. 打开 Electron 桌面端。
2. 展示标准 RAG 页面。
3. 导入一个 JSON / PDF / TXT 示例。

1. 切到 GraphRAG。
2. 进入图谱构建。
3. 导入语料。
4. 应用 Graph Schema。
5. 构建知识图谱。
6. 展示节点、边、统计和大图降级策略。
7. 进入图谱问答。
8. 提问一个跨实体或全局问题。
9. 展示 route、text evidence、graph evidence、global evidence。

### 10.3 私有语料复杂场景演示

1. 说明数据已经本地导出为结构化 JSON。
2. 说明系统保留联系人、会话、消息、时间、发送方等层级。
3. 点击“一键检测”。
4. 展示进度日志。
5. 展示系统如何区分普通 top-k 问答和全量联系人级分析。
6. 强调没有原文证据不会判断。

## 12. 当前边界与风险

| 风险 | 说明 | 应对 |
|---|---|---|
| 前端单文件过大 | `index.html` 承载过多逻辑 | 短期不重构，先补测试和文档；中期拆模块 |
| 大图渲染卡顿 | 节点和边过多会影响浏览器 | 大图跳过全量渲染，只显示统计和抽样 |
| LLM 配置依赖用户 | 无 API Key 时无法生成最终回答 | 前端必须明确提示缺少配置 |
| GraphRAG 证据不足 | 图谱实体不能替代原文证据 | 输出必须绑定 citation / message_id / chunk_id |
| 数据残留 | 旧集合或旧图谱可能污染问答 | 默认重建集合，导入前清理 runtime |
| 复杂问题漏召回 | top-k 不适合全量统计 | 引入分区扫描任务模式 |

## 13. 成熟度判断

如果以开源项目成熟度看，当前 PowerRAG 的状态可以这样判断：

- **功能广度**：较强。已经覆盖 RAG、GraphRAG、评估、桌面端。
- **工程稳定性**：中等。需要一键检查、CI、模块拆分和回归测试。
- **文档表达**：需要加强。能力很多，但需要更清晰的项目门面。
- **演示完整度**：较强。已有控制台和 Electron，但 Demo 路径需要固定。
- **研究价值**：较强。包含 GraphRAG、社区摘要、评估和复杂私有语料场景。
- **开源可读性**：中等偏上。结构有了，但 README、readiness、sample data 和验证命令仍需补齐。

结论：

> 这个项目不是从零开始的玩具项目，而是一个已经具备复杂 RAG / GraphRAG 能力的工程原型。短板主要在项目门面、稳定验证、流程解释和前端复杂度控制。只要补齐文档、健康检查、Demo 路径和证据链展示，它就能在短时间内呈现出接近成熟开源项目的面貌。

## 14. 推荐立刻补齐的四个文件

为了让项目更像一个成熟开源项目，最值得立刻补齐的是：

1. `README.md`：清晰 Quick Start 和能力说明。
2. `docs/OPEN_SOURCE_READINESS.md`：说明当前可演示状态、已知限制和验证命令。
3. `scripts/quick_health_check.ps1`：一键健康检查。
4. `tests/unit/test_open_source_readiness_contract.py`：保护 README、Electron、一键流程等关键承诺。

这四个文件不会改变架构，但能显著提高项目可信度。

## 15. 复盘结论

这次项目推进中最大的经验是：

1. RAG 系统不是“把数据塞进向量库”就结束，关键在于证据结构、召回路径和失败解释。
2. GraphRAG 不是可视化图谱，而是实体关系、社区摘要和全局检索共同构成的推理索引。
3. 对复杂私有语料，数据导出阶段就必须保留层级结构，否则后续问答无法补救。
4. 前端必须让用户看到系统正在做什么，否则长任务会被误解为卡死。
5. 成熟项目不只是功能多，还要能一键运行、一键检查、清晰介绍、明确边界。

PowerRAG 后续最重要的方向不是继续堆功能，而是把已有能力串成稳定、可解释、可验证的主线。
