# 今晚交流发言稿

## 30 秒版

我这边快速调研了一下 GraphRAG。它不是简单替代普通 RAG，而是在传统向量检索之外，引入知识图谱，把文档里的实体、关系、事件等结构化出来。它更适合多文档、多跳关系和全局归纳类问题，比如“这批文档的主要主题是什么”“某个实体和哪些事件有关”“跨文档有哪些隐藏关联”。

## 2 分钟版

GraphRAG 的典型流程是：文档切分、实体和关系抽取、构建知识图谱、实体去重和归一、图聚类或社区发现、生成社区摘要，最后查询时结合向量检索、图遍历和社区摘要。

相比普通 RAG，它的优势是能保留跨 chunk、跨文档的关系，适合回答更复杂的问题；缺点是建图成本高，实体和关系抽取质量不稳定，图谱更新和评估也更复杂。

工具上建议先看三条线：Microsoft GraphRAG 适合理解完整流程；Neo4j GraphRAG 适合落地到图数据库；LlamaIndex 或 LangChain 适合快速集成到已有 RAG 项目。如果要做 demo，可以先拿一小批文档，对比普通 RAG 和 GraphRAG 在复杂问题上的答案完整性、引用准确性、延迟和成本。

## 5 分钟版

我的初步结论是：GraphRAG 的价值主要体现在“关系”和“全局视角”上。普通 RAG 往往是从问题出发，找最相似的几个文本块，然后交给 LLM 总结；但如果问题涉及多个实体、多个文件或者整体主题，单纯相似度检索可能召回不完整。

GraphRAG 会先离线构建一个图结构。图里一般有 Document、Chunk、Entity、Relationship、Community 等节点或数据表。实体和关系来自 LLM 或信息抽取模型，community summary 则用于回答全局性问题。查询阶段可以分成 global search、local/entity search、graph traversal、vector+graph hybrid 等方式。

但它也有明显风险：实体重复、关系幻觉、schema 不稳定、跨 chunk 关系丢失、增量更新成本高。如果我们的数据量不大、问题比较简单，直接上 GraphRAG 可能投入产出比不高。所以我建议先做一个小 POC，不要一开始就追求完整系统。

## 可以主动抛出的问题

1. 我们的场景更偏“问单个文件细节”，还是“跨文档综合分析”？
2. 是否已经有比较稳定的实体类型和关系类型？
3. 评估指标用什么：答案完整性、引用准确性、成本、延迟，还是人工主观评分？
4. demo 先用 Microsoft GraphRAG 还是 Neo4j GraphRAG？
