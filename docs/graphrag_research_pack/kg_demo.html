Same Corpus, Three Graphs

同一批文档，三条工具路线会长成三种“图”。

下面不是把知识图谱当图片看，而是演示三种系统各自会关心什么数据结构：Microsoft GraphRAG 关心社区摘要和全局/局部检索，Neo4j GraphRAG 关心可落库的节点关系 schema，LlamaIndex / LangChain 关心快速把文档变成可查询的图索引。

选择视角

实体 / 设备
故障 / 原因

Microsoft GraphRAG：社区摘要视角

先从文档抽取实体和关系，再形成实体社区与社区报告；查询时可以走 global search 或 local/entity search。

演示问题：FJ-17 齿轮箱过温可能由什么导致，建议怎么处理？

      三条线的差异
      
        
      
          
      
            
      Microsoft
      
            
      更像“完整 GraphRAG 方法论”：索引阶段抽实体关系并生成社区摘要，查询阶段区分全局和局部检索。
      
          
      
          
      
            
      Neo4j
      
            
      更像“工程落地形态”：节点、边、属性都进图数据库，可以用 Cypher 做精确路径查询和可视化。
      
          
      
          
      
            
      LlamaIndex / LangChain
      
            
      更像“快速集成方式”：从现有 RAG pipeline 里抽三元组或 property graph，先跑起来验证效果。
