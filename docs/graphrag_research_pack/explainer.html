概念解释手册

把知识图谱和 GraphRAG 的关系讲清楚。

这页负责解释概念：知识图谱不是一张图片，而是实体、关系和属性组成的数据结构；GraphRAG 则是利用这种结构来增强检索和生成的 RAG 方法。

今晚发言稿 调研大纲 三种图谱演示 Microsoft 论文 本地资料入口
Document 资料不再只是碎片，而是可以追溯来源的知识单元。
Entity 人、组织、设备、事件和概念被抽取成可连接节点。
Relation 关系让模型看到跨 chunk、跨文档的隐含路径。
Community 社区摘要承担全局归纳，适合回答主题和趋势问题。
01 Verdict

先讲边界，再讲方案。

GraphRAG 的投入比普通 RAG 更高，所以调研重点不是“更先进”，而是“什么时候值得”。

一句话判断

如果项目目标是让模型读几份文档回答具体问题，普通 RAG 足够；如果目标是理解一批资料中的实体关系、主题结构和跨文档关联，GraphRAG 值得做 POC。

适合

  • 多文档综合分析
  • 多跳关系查询
  • 实体关联追踪
  • 全局主题归纳

谨慎

  • 单篇文档事实问答
  • 关系结构不明显的小数据集
  • 低延迟、低成本的简单问答
  • 缺少评估集的直接落地
02 Pipeline

从文档到答案的九步。

GraphRAG 最核心的变化，是先把知识结构组织出来，再按结构检索和聚合。

01文档加载PDF、网页、Markdown、业务文档进入资料池。
02文档切分切成 chunk，并保留来源、页码和上下文。
03实体抽取抽取人、组织、设备、事件、概念等节点。
04关系抽取识别属于、导致、影响、依赖、提到等关系。
05图谱构建把 document、chunk、entity、relationship 组织为图。
06实体归一合并别名、同义词、大小写差异和重复节点。
07社区发现找出相关实体群组，形成更高层级的主题结构。
08社区摘要为实体群组生成可检索、可引用的摘要。
09混合查询结合向量检索、图遍历、实体检索和摘要生成答案。
03 Talk Track

今晚可以直接这样讲。

按时间选择版本，越往后越适合展开讨论。

我这边快速调研了一下 GraphRAG。它不是简单替代普通 RAG，而是在传统向量检索之外，引入知识图谱，把文档里的实体、关系、事件等结构化出来。

它更适合多文档、多跳关系和全局归纳类问题，比如“这批文档的主要主题是什么”“某个实体和哪些事件有关”“跨文档有哪些隐藏关联”。

GraphRAG 的典型流程是：文档切分、实体和关系抽取、构建知识图谱、实体去重和归一、图聚类或社区发现、生成社区摘要，最后查询时结合向量检索、图遍历和社区摘要。

相比普通 RAG，它的优势是能保留跨 chunk、跨文档的关系，适合回答更复杂的问题；缺点是建图成本高，实体和关系抽取质量不稳定，图谱更新和评估也更复杂。

工具上建议先看三条线：Microsoft GraphRAG 适合理解完整流程；Neo4j GraphRAG 适合落地到图数据库；LlamaIndex 或 LangChain 适合快速集成到已有 RAG 项目。

我的初步结论是：GraphRAG 的价值主要体现在“关系”和“全局视角”上。普通 RAG 往往是从问题出发，找最相似的几个文本块，然后交给 LLM 总结；但如果问题涉及多个实体、多个文件或者整体主题，单纯相似度检索可能召回不完整。

GraphRAG 会先离线构建一个图结构。图里一般有 Document、Chunk、Entity、Relationship、Community 等节点或数据表。实体和关系来自 LLM 或信息抽取模型，community summary 则用于回答全局性问题。

但它也有明显风险：实体重复、关系幻觉、schema 不稳定、跨 chunk 关系丢失、增量更新成本高。所以建议先做小 POC，不要一开始追求完整系统。

场景我们更偏问单个文件细节，还是跨文档综合分析？
Schema是否已有稳定的实体类型和关系类型？
评估看答案完整性、引用准确性、成本、延迟，还是人工评分？
Demo先用 Microsoft GraphRAG，还是 Neo4j GraphRAG？
04 Tools

工具路线不要铺太散。

先抓住完整流程、图数据库落地和快速集成三条主线，其他工具作为补充。

Microsoft GraphRAG

完整流程社区摘要Global Search

最适合理解标准 GraphRAG 为什么这样设计，包含 indexing、query、community report、local/global/DRIFT search。

建议：首选调研对象。

Neo4j GraphRAG

图数据库CypherKG Builder

适合把 Document、Chunk、Entity、Relationship 存进 Neo4j，方便查询、可视化和工程落地。

建议：首选落地对象。

LlamaIndex PropertyGraphIndex

Python POCProperty Graph

和 LlamaIndex ingestion/query 体系结合较好，适合快速验证图谱索引和查询。

建议：适合时间紧时做轻量 demo。

LangChain + Neo4j

LLMGraphTransformer集成灵活

适合已有 LangChain 项目，用 LLM 生成图文档并写入 Neo4j。

风险：质量高度依赖 prompt、模型和 schema 约束。

LightRAG

轻量速度

强调 simple and fast 的 RAG 变体，和 Microsoft GraphRAG 思路不完全相同。

建议：作为扩展调研。

Graphiti

Temporal KGAgent Memory

面向动态、时间感知知识图谱，适合 agent memory 和持续变化的数据。

建议：作为高级方向。

spaCy

NER依存句法规则

工程稳定、可训练、可加规则，适合实体预处理，但不是完整 GraphRAG 框架。

Stanford CoreNLP OpenIE

三元组传统抽取

不依赖 LLM，可作为 baseline；语义质量和领域适配需要单独评估。

REBEL

关系抽取模型对照

端到端抽取关系三元组，英文更合适，可作为 LLM 抽取效果的对照。

05 POC

用小数据集先验真价值。

GraphRAG 的价值要靠复杂问题体现，POC 应该直接对比普通 RAG 和 GraphRAG。

一

跑通 Microsoft GraphRAG

重点看 entities、relationships、community reports 的输出质量。

二

落库到 Neo4j

观察图结构是否可查、可视化、可解释，是否能支持业务问题。

三

和普通 RAG 对比

如果复杂问题提升不明显，回到 hybrid RAG；如果提升明显，再投入 schema 和评估。

评估维度 观察点
答案完整性是否覆盖问题多个方面，尤其是跨文档证据。
引用准确性是否能回到原文来源，而不是只给抽象结论。
关系可信度实体和关系是否真实来自文本，是否存在幻觉关系。
成本与延迟建图 token 成本、查询 token 成本、索引时间和查询时间。
维护成本新增文档后是否容易更新，社区摘要是否需要重算。
