# 命名重构说明

这次不是把所有底层工程文件强行改名，而是给项目加了一层“人能看懂的成果目录”。

## 新旧对应

| 你应该看的新目录 | 原来的工程位置 | 含义 |
| --- | --- | --- |
| `01_资料输入_14本PDF和问答JSON` | `data_pipeline/raw/tsinghua_gas_turbine_books/` | 老师给的资料、问答 JSON、下载记录 |
| `02_OCR结果_13本扫描PDF` | `data_pipeline/ocr/tsinghua_gas_turbine_books/` | 13 本扫描 PDF 的 OCR 文本 |
| `03_普通RAG数据库_14本资料` | `storage_layer/runtime/ocr_enriched_rag_chroma/` | 普通 RAG 的 ChromaDB 数据库 |
| `04_检索测试结果_10个问题` | `evaluation/reports/ocr_enriched/` | 用 10 个问题测试检索效果 |
| `05_知识图谱POC_三元组和人工判断` | `kg_pipeline/poc/labelstudio_course_kg/` | 知识图谱 POC、schema、三元组、人工判断 |
| `06_汇报材料_发群和组会` | `docs/` | 给老师看的阶段总结材料 |

## 为什么不直接把数据库文件夹改成中文

OCR 文本、报告、图片可以放中文目录；但是 ChromaDB 数据库本体不能随便改成中文路径。已经验证过，中文路径会让 hnswlib 只写出部分索引文件，重启后报错。因此：

- 人看的入口：中文目录。
- 程序跑的数据库：继续保留英文路径。
- 汇报时只说“普通 RAG 数据库已建成”，不用说 Chroma 的底层文件名。
