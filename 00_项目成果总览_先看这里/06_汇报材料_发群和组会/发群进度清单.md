# 知识图谱 / GraphRAG 方向进展清单

可直接发群版本：

```text
各位老师、同学，目前知识图谱 / GraphRAG 方向这几天的框架检索、尝试和阶段性产出如下：

- 先把方向重新收敛了一下：我现在理解 GraphRAG 不是直接做一张“图的图片”，而是先从文档里构建可检查的知识图谱，再用图结构辅助后续检索和问答。因此当前重点先放在 Graph construction，也就是实体、关系、证据和人工评审这一条链路。

- 框架方面主要看了三条线：Microsoft GraphRAG 用来理解完整流程，尤其是 Document / TextUnit / Entity / Relationship / Community Report；Neo4j / KG Builder 更适合后续图数据库落库和查询；LlamaIndex / LangChain 更适合接到现有 Python RAG 工程里做轻量 POC。当前判断是不从零重做 Microsoft GraphRAG，而是在已有 RAG 流程后面补一条图构建支线。

- 这次已经基于之前的 Label Studio 导出 JSON 做了一个可展示的 POC：输入是 `project-1-at-2026-04-09-07-02-f7d8cb93.json`，从中抽取 OCR 文本块，再组织 chunk，设计课程知识体系 schema，生成候选三元组，并为每条三元组绑定 evidence。

- POC 当前结果：原始 JSON 共 4 页，抽出 246 个 OCR 文本块，组织成 14 个 chunk；schema 中设计了 10 类实体和 15 类关系；生成 27 条候选三元组，27 条通过 schema 校验。

- 我还做了一个可操作的人工评审工作台，不只是静态 HTML 展示。页面里可以逐条查看三元组、对应 evidence 和所在 chunk，然后人工标记“对 / 错 / 待讨论”，最后导出 JSON/CSV 评审结果。

- 已完成一轮人工评审：27 条候选三元组中，26 条判断为正确，1 条标记为待讨论，0 条明确错误。待讨论项是“课程是否使用课程知识图谱”这条关系，主要问题不是实体，而是关系粒度是否表达得足够准确。

- 基于这轮评审，我也生成了一张知识图谱图片作为展示产物：图中节点表示课程、机构、方法、维度、模块、知识点、应用场景和效果；边表示建设方、采用方法、包含维度、理论基础、方法支撑、应用于等关系；绿色边表示人工判断正确，橙色边表示待讨论关系。

- 这次探索中最大的认识是：实体其实相对容易找，真正难的是关系粒度。关系太泛，比如 related_to，后续检索没有语义价值；关系太细，比如把中文谓语逐个翻译成英文关系，又会导致关系爆炸、难以复用。所以后续 schema 需要保持中等粒度，让每类关系能回答一个稳定问题，例如“哪个模块支撑哪个模块”“知识图谱应用在哪些教学环节”。

- 当前能说明的是：已经完成了从真实 JSON 输入到图构建、evidence 展示、人工评审和知识图谱可视化的最小闭环。当前还不能说完整 GraphRAG 问答、Neo4j 落库或 LLM 自动抽取质量已经完成。

- 下一步准备：用同一份 JSON 和同一套 schema 让 LLM 自动抽取三元组，再与这次人工评审结果对比，统计实体错误、关系错误、方向错误、evidence 缺失和遗漏关系；同时根据待讨论关系继续调整 schema 的关系粒度。
```
