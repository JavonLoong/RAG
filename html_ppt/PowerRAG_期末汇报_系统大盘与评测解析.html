🛠️ 系统核心子系统与职责定义 8 大模块
  • 1. 用户界面与外壳 (UI & Shell)

    基于 Electron 桥接本地 API。前端使用 D3.js 渲染拓扑图，提供多级 OCR 预览和评测看板。

  • 2. 后端服务网关 (FastAPI Gateway)

    提供 /api/files、/api/query、/api/graph 和 /api/evaluation 等高内聚接口路由。

  • 3. 数据治理清洗管线 (Data Ingestion)

    支持 PDF / JSON 多通道提取，清洗转换噪音，根据大纲与语义断点进行物理分块。

  • 4. GraphRAG 构建管线 (KG Ingestion)

    通过 LLM 进行强 Schema 约束提取，将实体三元组严格绑定原文证据，利用 Leiden 算法执行多级社群划分与摘要。

  • 5. 混合检索与编排引擎 (Orchestration)

    基于大模型判断 Query 类型，路由至不同底层链路（混合检索、Local/Global 检索、分区扫描）。

  • 6. 持久化数据层 (Storage Layer)

    ChromaDB 隔离保存分块向量，NetworkX 本地维护图扑结构，免去部署外部独立图库的开销。

  • 7. 模型适配器层 (Adapters)

    支持 OpenAI 兼容 API 以及本地部署 BGE-M3 双向向量模型，屏蔽供应商底层差异。

  • 8. 评估回归层 (QA & Evaluation)

    针对 47 题测试集自动化遍历打分，并将错误归入 retrieval_empty、llm_hallucination 等状态，驱动持续回归。

🔄 核心业务数据流 流转路径

路径 A：多路径图索引构建流

1

文档输入与 OCR 提取

扫描件流式加载，运行 RapidOCR 输出带位置信息的文本。

2

物理分块与双向处理

向量方向入库 ChromaDB；图谱方向由大模型抽取三元组，利用证据绑定器反向挂载源段落页码。

3

Leiden 聚类与社群摘要

生成具有拓扑结构的关系，将关系做多层 Leiden 划分并生成高层级文本摘要用于宏观语义索引。

路径 B：意图路由检索问答流

1

用户 Query 解析与路由

大模型意图路由分类：局部事实、实体关系、全局宏观或会话批量扫描。

2

混合检索或图提取

召回向量 Chunk、BM25 词频匹配或 Local / Global GraphRAG 关系链检索。

3

RRF 融合与大模型生成

排重去重，装填携带严格 Grounding 指令的 Prompt，输出带Evidence链接的可追溯答案。

以下内容整理自汇报演讲稿中的核心防守策略，点击展开后可直接查看推荐的回答口径和逻辑链，帮助您在面对老师提问时应对自如：

❓ 问题一：你们的 PowerRAG 架构与普通的问答 Demo (如网页直接接大模型) 有何本质区别？ ▼

核心答辩策略：强调“数据工程”、“可信度”与“评测闭环”。普通的 Demo 属于“前端连上 LLM 就算通”，而 PowerRAG 是一套完整的可信工程系统：

  • 资料入口治理：专门设计了 JSON 结构解析、文字 PDF 抽取以及扫描 PDF 降噪 OCR 管道，解决了真实资料不干净的问题。
  • 证据链溯源 (Grounding)：每一个 Chunk 在入库时均利用证据绑定器挂载了 [文件名, 页码, 标题]，回答时会强制高亮输出 Evidence Links，防止大模型捏造事实。
  • 评测即开发工具：拥有本地 47 题专家评测集和外部对标评测框架，通过失败原因分类（Triage）归纳错误，以数据分数指导检索策略的改进。
❓ 问题二：在目前的评测中，为什么 Keyword (Recall@K = 0.833) 表现优于 Dense (0.583) 向量检索？ ▼

核心答辩策略：指出“专业领域术语特点”与“Embedding 表征局限”。

“这是由我们动力装备专业语料的特点决定的。动力系统资料里含有大量极度明确的强专有名词、设备名、部件型号和参数代码（如‘压气机’、‘联合循环’、‘Goldwind’等）。这些强术语是极其明确的物理锚点，在关键词检索（BM25）中能被 100% 召回。”

而普通的向量模型（Dense Hashing）由于没有在能源动力垂域语料上做精调（Fine-tuning），无法准确表征这些特定符号在高维空间中的语义关联，因此召回效果受到影响。这更说明了我们开发评测平台的必要性——只有评测能揭示真实表现，防止技术盲目堆叠。我们下一阶段的优化重点就是引入更好的 Embedding 模型和 Reranker。

❓ 问题三：你们引入了 GraphRAG，它到底解决了普通 RAG 无法解决的什么问题？ ▼

核心答辩策略：强调“多步关系追踪”与“跨文档全局概括”。

  • Local GraphRAG (关系追踪)：针对例如“某故障可能关联哪些部件，以及会导致什么工况变化”的逻辑链条，普通 RAG 碎片检索很难拼出完整的因果图谱。GraphRAG 可以沿着实体节点和一/二阶关系边追踪拓扑关系，拼接出清晰的关系证据链。
  • Global GraphRAG (全局综合)：面对“比较几本手册中的维护规律”这种宏观问题，普通 RAG 的 Top-k 检索只会定位到相似段落，容易遗漏整体数据。GraphRAG 通过 Leiden 社群算法，将网络切分为局部社群并预先生成社区摘要（Community Summary），从而提供了宏观索引。

边界说明：GraphRAG 在系统里是作为“关系型与全局型问题的增强路线”，而不是替代普通事实检索的唯一路线。简单事实问题系统依然会路由走开销更小的标准 RAG。

❓ 问题四：在外部评测中得到了 93.47 分，这能代表你们的系统在清华燃机知识库上已经完全通过验收吗？ ▼

核心答辩策略：明确区分“外部 Retrieval-first Benchmark”和“本地动力装备专家 Set”。

“不能混淆。93.47 分是通过接入下载至本地的公开评测集（包括 50 道 legal-rag-bench 题和 50 道 ragbench-hotpotqa 题）测出来的。它通过了我们设定的‘Open Source 90 外部质量门禁’。”

这个分数的意义在于：验证了我们这套数据解析、切分、召回和 Rerank 的工程管线在通用检索任务上已经达到了业界成熟系统的水平。但这不代表它就是动力装备垂域的最终专家验收。针对清华燃气轮机知识库的真实验收，我们需要依靠后续扩充的 100+ 题本地专家 Gold Set，配合 equipment_type 和 scenario_type 新增字段，来建立完全贴合设备故障场景的验收矩阵。

❓ 问题五：评测平台所说的“失败归因”和“回归测试”是如何在开发中发挥作用的？ ▼

评测如果不做原因分类，对开发就没有意义。因此我们设计了归因分析器 (Triage)，将测试案例分类为：

  • retrieval_empty：没有检索到有用段落（说明需要改分块或增强召回）。
  • llm_hallucination：检索到了证据但 LLM 回答时发生篡改（说明要修改 Prompt 限制条件）。
  • router_error：分类路由走错了路径（比如事实题走到了全局检索）。

每次我们修改代码或配置，都会自动跑回归测试，确认这 47 道题目的打分和归因是否改善，用客观指标指导迭代方向，保证修改不引起旧问题的退化。

❓ 问题六：项目目前处于什么开发阶段？后续最重要的改进计划是什么？ ▼

当前我们已经完成了第一阶段的完整工程闭环（实现了资料统一入口、带元数据证据构建、多模态检索对比、GraphRAG 局部/全局基础架构，以及本地 47 题评测闭环）。

下一阶段（也是向正式验收推进的节点）的核心计划包括：

  1. 接入专用 Embedding 与重排模型：利用垂域语料精调或部署更强的多语言 BGE-M3 模型以提振语义检索性能。
  2. GraphRAG 同题对比测试：在 47 题库中抽选关系型问题，对比 Local、Global 和 RAG 纯事实检索的效果，划清技术应用边界。
  3. 扩展验收集字段：补齐 equipment_type 和 scenario_type 两个维度，建立严密的专家评测 Gold Set，从课程交付迈向高可靠性的可复现实验平台。
