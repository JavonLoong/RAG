# RAG / GraphRAG 项目探索过程审查报告

> 审查时间：2026-06-20  
> 审查对象：本地 Git 历史、GitHub 远端、GitLab 远端、项目 README / TECH_STACK / evaluation / html_ppt / docs 交付材料。  
> 审查目的：不是再罗列最终成果，而是还原这个项目从“零散页面和原型”走到“系统工程与评测闭环”的探索路径，明确走过的弯路、修正动作，以及它是否回到最初的问题。

## 0. 先给结论

这个项目真正值得讲的，不是“我们最后做了一个 RAG 网页”，而是：

> 我们一开始并不知道最终形态应该是什么，于是从 JSON 解析、Chroma 检索、前端页面、OCR、GraphRAG POC、评测集、失败分析、汇报包装一路试错。最后通过 Git 历史可以看到，团队逐渐把方向从“功能堆叠和演示包装”拉回到“动力装备资料的可信问答闭环”：数据能进来，证据能检索，图谱能解释，评测能比较，失败能归因，演示能复现。

所以汇报时要讲“探索过程”，不是只讲“最终成果”。最终成果只是最后一层，真正的价值在于中间的判断和矫正。

走的弯路：

- Tesseract.js 在中文扫描件上质量不够。
- 浏览器端 OCR 受模型加载、file:// 协议、性能和兼容性限制。
- 一开始容易把问题理解成“换一个 OCR 引擎就行”，但实际还需要服务启动、状态反馈、文本预览、缓存、导出和错误恢复。

矫正动作：

- 从单一 OCR 引擎改为多引擎试错。
- 从纯浏览器能力改为本地 OCR server + 前端状态感知。
- 从“识别完就算了”改为“识别文本可预览、可导出、可入库、可复用”。

汇报表达：

> OCR 这条线说明我们面对的不是干净 benchmark，而是真实工程资料。我们走过多引擎试错，最后认识到 RAG 的上限首先受资料质量影响。

## 4. 我们和别人“做一个 RAG 页面”的区别

别人常见做法：

1. 准备几个文档。
2. 丢进向量库。
3. 做一个聊天框。
4. 问几个问题，回答能出来就结束。

我们的路线不同：

1. 先处理真实资料进入系统的问题：JSON、PDF、OCR、扫描件、导出文本。
2. 不只做向量检索，还做 keyword、dense、hybrid、graph、reranker 的比较。
3. 不只画知识图谱，而是强调 schema、实体关系、evidence 绑定和 GraphRAG 子集。
4. 不只说“效果不错”，而是用 60 题评测集、baseline 对比和失败归因。
5. 不只写代码，还做可运行控制台、HTML 演示、GitHub Pages 评测入口。
6. 不只包装成果，还在历史里能看到主动清理偏航材料、回到课程目标。

一句话：

> 我们不是把 RAG 当成一个聊天组件，而是把它当成“真实工程资料如何变成可信答案”的完整链路来做。

## 5. 这个项目的意义到底在哪里

这个项目的意义不在于“我们用了很多技术名词”，而在于它回答了一个课程项目里很容易被忽略的问题：

> 当资料很乱、来源很多、内容有 OCR 错误、术语很强、问题需要跨文档理解时，怎样让一个问答系统的答案变得可追溯、可评测、可改进？

对应到最终系统，意义有五层：

### 5.1 数据意义

它不是直接拿干净文本做 demo，而是处理真实资料接入、OCR、分块、缓存、导出和入库。

### 5.2 检索意义

评测结果显示 keyword 目前强于或接近 hybrid，这说明动力装备资料有强术语特征。这个结论本身很有价值，因为它阻止我们盲目相信“向量一定更高级”。

### 5.3 GraphRAG 意义

GraphRAG 的意义不是把图画出来，而是处理实体、关系、跨文档解释和全局摘要问题。现在最正确的表达是：

> GraphRAG 已经完成构建链路和同题子集设计，下一步要在 local/global 上做同题实测。

### 5.4 评测意义

评测让项目从“能跑”变成“可比较”。失败案例不是扣分项，而是系统优化路线的来源。

### 5.5 汇报意义

HTML PPT 和 GitHub Pages 让复杂工程可以被快速理解；它不是替代系统，而是系统的讲解入口。

## 6. 是否回到初心

我认为：回到了，但不是一开始就回到，而是绕了一圈才回到。

最初初心可以定义为：

> 为动力装备资料建立一个可信、可追溯、可评测的 RAG / GraphRAG 知识系统。

偏离初心的阶段：

- 过度堆前端功能。
- 过度追求 OCR 引擎切换。
- 过度进入 Challenge Cup / 答辩包装材料。

回归初心的证据：

- 清理 Challenge Cup 材料。
- 保留 60 题评测、baseline、失败分析、GraphRAG 子集。
- 修复 Chroma 和启动脚本。
- 把展示页和评测平台做成系统入口，而不是把它当成全部成果。

所以汇报里可以这样说：

> 这个项目不是一条直线做出来的。我们先从能跑的页面开始，后来发现页面不能代表可信系统；又从 OCR、图谱和评测里逐步找到真正的问题；中间也有过偏向包装的阶段，但最后我们把材料清理掉，回到“资料如何变成可信答案”这个核心问题。

这句话会比“我们做了一个 RAG 系统”高级很多。

> 如果只看最后的页面，可能会觉得这是一个 RAG 展示系统。但从 Git 历史看，它其实是一个被真实问题一步步逼出来的系统。最早我们只是想把 JSON 数据解析、分块、入 Chroma，然后能查回来；后来发现资料不全是干净文本，所以开始做 PDF、OCR、扫描件和导出；再后来发现页面上能问答不代表结果可信，于是做了 keyword、dense、hybrid 的 baseline，对 60 题评测集做同题比较；再往后发现 GraphRAG 不能只画图，必须绑定 evidence、筛出适合图谱的问题子集；中间我们也走过偏向答辩包装的路，但后来把 Challenge Cup 材料清理掉，回到课程 RAG 项目本身。  
>   
> 所以我今天想讲的不是“我们最后做了一个网页”，而是“我们怎么从一个零散页面，逐步收敛成一个围绕动力装备资料的可信问答系统”。它和普通 RAG demo 的区别在于：它有真实资料接入，有多路检索，有 GraphRAG 构建链路，有评测集，有 baseline，有失败归因，也有可演示的网页入口。这个过程本身，才是项目最核心的价值。
