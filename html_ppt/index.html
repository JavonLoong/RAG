Problem

03 / 13

工程资料多、格式杂、追溯难。

动力装备领域资料包含扫描 PDF、维修报告、教材、JSON 标注和结构化表格。直接交给大模型容易出现三类问题：证据找不到、回答不可追溯、跨文档关系无法组织。

资料入口复杂

扫描件、DOCX、CSV、Markdown 和 Label Studio JSON 需要统一解析。

术语召回要求高

燃气轮机、部件、工况、故障、参数等词汇不能只依赖模糊语义。

答案必须有证据

面向课程汇报和后续论文实验，答案要回到原始 chunk 或图谱证据。

核心判断：这不是“聊天页面”，而是工程资料治理和可信问答链路。

Positioning

项目价值不是单点算法，而是把资料处理、检索、图谱、评测和前端展示串成可运行闭环。
适合汇报表达：工程闭环、可复现、可继续扩展。

Data Pipeline

06 / 13

数据接入不是上传文件，而是把资料变成可信证据。

  • 本地 OCR 服务器优先处理扫描 PDF，浏览器端 OCR 作为兜底。
  • 解析结果保留页码、文件来源、类型、chunk 参数和处理日志。
  • 文件队列区分待处理、已处理、失败、跳过，支持导出备份。
  • 通过集合隔离不同数据源，避免重复覆盖和实验污染。

Retrieval

07 / 13

普通 RAG 是稳定 baseline，混合检索是后续质量提升入口。

Dense 向量召回

把问题映射到 embedding 空间，召回语义相近的 chunk，适合概念、定义、流程类问题。

BM25 关键词召回

对强术语、设备型号、参数名更稳，避免向量模型把精确词汇稀释掉。

Hybrid / RRF 融合

将多路召回归一化合并，再用评测集分析哪些题型适合融合，哪些题型需要路由。

答辩口径：当前普通 RAG 链路已经能跑，GraphRAG 和混合检索用于解决跨文档、跨实体、解释结构更复杂的问题。
避免说“GraphRAG 一定更好”，改说“需要同题对比和消融”。

GraphRAG

08 / 13

GraphRAG 当前重点是 construction 与 evidence，而不是空泛画图。

已完成的价值在于：从真实 JSON / OCR 文本抽取实体关系，绑定 evidence，人工判断三元组质量，并把图谱构建链路纳入后续问答和评测路线。

已经可以讲

Schema 约束、候选三元组、证据绑定、人工评审、SQLite 图存储、GraphRAG 子集问题识别。

不要过度声称

不要说完整在线 GraphRAG 问答已经最终完成；当前更准确说法是“构建链路与 context-only 编排已跑通”。

这个边界能保护项目可信度。
