# RAG / GraphRAG 期末汇报作战稿

> 当前日期：2026-06-20。正式汇报按 2026-06-22 左右准备。目标不是“把所有工作讲完”，而是让老师在有限时间内相信：这个项目有真实工程量、有评测闭环、有问题意识、有继续优化路线。

## 一句话总定位

我们做的不是一个普通聊天页面，而是一个面向动力装备资料的 RAG / GraphRAG 知识平台：从资料处理、索引构建、混合检索、知识图谱构建，到评测集、baseline 对比、失败归因和网页端演示，形成了一个可复现的工程闭环。

## 8-10 分钟汇报结构

### 0:00-0:40 开场：先把项目价值讲清楚

不要从“我用了什么技术”开头。先说问题：

> 动力装备资料通常包含教材、维修报告、OCR 文本、术语和跨文档关系。如果只做一个简单问答页面，模型很容易检索不到关键证据，或者生成没有来源的答案。所以我这个项目重点不是单次问答，而是把资料处理、检索、图谱和评测串成一个可信问答系统。

这一段的目标：让老师知道你不是做了一个壳，而是在解决“资料可信问答”的问题。

### 0:40-2:00 系统总览：展示 HTML PPT 第 1-2 页动态链路

讲法：

> 这里我用动态链路展示一次问题从输入到答案输出的过程。系统先经过资料接入和 OCR/JSON 解析，然后进入 Chunk、向量库、关键词索引和图谱构建。在线问答时，问题会走检索、上下文打包、答案生成和证据引用。旁边的运行追踪表示我不是只做页面，而是把每一步都设计成可以观测、可以定位、可以评测。

重点词：

- 数据入口
- 索引构建
- 混合检索
- GraphRAG 扩展
- 证据引用
- 运行追踪

### 2:00-3:20 架构页：讲模块化，不要陷入代码细节

讲法：

> 项目按工程模块拆成 data_pipeline、retrieval_engine、kg_pipeline、rag_orchestrator、evaluation、api_server 和 frontend_app。这样拆的原因是每一层职责独立：数据管线负责资料变成可检索文本，检索层负责召回证据，图谱层负责实体关系和社区摘要，编排层负责把证据转成答案，评测层负责判断结果好坏。

老师可能听到的高分点：

- 不是脚本堆叠，是分层系统。
- 不是只跑 Demo，是有评测模块。
- 不是只说 GraphRAG，而是先把构建链路和边界讲清楚。

### 3:20-4:40 检索与 baseline：把数字讲得像实验

引用 Day3 baseline：

| 方法 | Question recall@K | Avg keyword coverage | 结论 |
| --- | ---: | ---: | --- |
| keyword | 0.833 | 0.563 | 当前最强，说明动力装备资料强术语很多 |
| dense_hashing | 0.583 | 0.300 | 可复现语义基线，不代表最终 embedding 上限 |
| hybrid_rrf | 0.817 | 0.520 | 接近 keyword，但还需要路由和 reranker |

讲法：

> 我没有只说“检索效果还可以”，而是做了 60 题评测集和三类 baseline 对比。结果显示 keyword 在当前语料里反而最强，说明动力装备资料有很多显式术语和设备名；hybrid_rrf 接近 keyword，但没有稳定超过它，所以后续不能简单融合，而要做问题类型路由和 reranker。

这一段非常重要：它体现你有实验意识，而不是堆技术词。

### 4:40-5:50 失败归因：主动承认问题，反而更像高质量项目

讲法：

> 我们还做了失败案例分析。失败不是一句“效果不好”，而是分成 hybrid_dilution、partial_ranking_gap、evaluation_concept_gap 等类别。比如有些问题 keyword 能命中强术语，但 hybrid 融合以后反而稀释；有些问题召回到了相关材料，但排序没有把关键证据放到前面；还有些问题是评测概念没有完全覆盖。

高分表达：

> 失败分析的价值是把系统缺陷转化成下一步可执行任务：source scope、reranker、embedding 替换、GraphRAG 子集实测。

### 5:50-7:00 GraphRAG：讲边界，避免吹过头

引用 GraphRAG 子集：

- 60 题中筛出 10 题 GraphRAG 子集。
- context 题 8 个，global 题 4 个。
- 当前文本 baseline 最优覆盖率均值 0.633。
- 6 个题优先进入下一轮 GraphRAG 实测。

讲法：

> GraphRAG 这部分我不会说它已经在所有题上超过普通 RAG。更准确的说法是：我已经完成了知识图谱构建链路，包括 schema 约束、实体关系、evidence 绑定、人工评审和同题子集识别。GraphRAG 更适合跨文档综合、实体关系解释和全局摘要类问题，所以我从 60 题中筛出了固定子集，作为后续 local/global GraphRAG 对比的基准。

这段一定要稳。不要说“完整商业级 GraphRAG 已完成”。

## 老师可能追问与回答

### Q1：你这个和普通 ChatGPT 问答有什么区别？

答：

> ChatGPT 是通用生成模型，我这里重点是私有资料的可信问答。系统会先把动力装备资料处理成可检索语料，再通过检索、图谱和证据引用控制答案来源，并用评测集验证召回效果。

### Q2：为什么 keyword 比 hybrid 还高？是不是你的 hybrid 没做好？

答：

> 这正是评测的价值。当前资料中强术语、设备名和报告词很多，所以 keyword 很强。hybrid 接近 keyword 但没有稳定超过，说明下一步不能简单融合，而要做查询类型路由、reranker 和更正式的 embedding。

### Q3：GraphRAG 到底完成了什么？

答：

> 完成的是构建链路和评测准备：schema 约束、实体关系抽取、evidence 绑定、人工评审、图谱存储、GraphRAG 同题子集筛选。在线 GraphRAG 问答还需要继续做 local/global 同题实测，所以我不会夸大成已经全面超过 baseline。

### Q4：你的评测系统有什么意义？

答：

> 它把项目从“能跑”变成“可比较”。我可以看到不同检索策略在同一批问题上的 recall、coverage、强弱失败案例，并把失败原因转化成下一步优化任务。

### Q5：如果现场网络不稳定怎么办？

答：

> GitHub Pages 页面可以提前打开；评测平台遇到 GitHub API 限流时会使用内置快照继续演示。同时我也准备了离线截图和 Markdown 报告，不依赖现场重新跑完整评测。
