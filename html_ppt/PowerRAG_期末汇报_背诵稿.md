# PowerRAG / GraphRAG 期末汇报背诵稿

> 用法：先完整背“正式背诵稿”。括号里的“停顿”不用念，是提醒自己换气和切页。  
> 核心目标：不要讲成“我做了一个网页”，而要讲成“我把动力装备资料变成了可检索、可引用、可评测的可信问答系统”。

## 一句话总定位

PowerRAG 是一个面向动力装备知识库的检索增强生成系统。它不是把问题直接丢给大模型，而是先把 PDF、扫描件、JSON 标注和维修报告等真实资料整理进系统，再通过检索、图谱和评测，让答案能够找到证据、回到来源，并持续优化。

## 正式背诵稿

### 1. 开场：PowerRAG 是什么

各位老师好，我汇报的项目叫 PowerRAG。  

从名字上看，Power 指的是动力装备知识库，RAG 指的是检索增强生成。也就是说，这个系统的核心不是让大模型凭自己的知识直接回答，而是让它先从我们已有的动力装备资料里检索证据，再基于证据生成答案。

我对 RAG 的理解是：检索的目的，是增强生成的可信度。一个答案如果只是模型自己说出来的，我们很难判断它是不是来自资料；但如果它能对应到具体资料、页码、证据块和来源，那么它就不只是一个回答，而是一个可以被检查的工程结果。

所以这个项目最后想做的，不只是一个问答页面，而是一个面向动力装备资料的可信问答与评测平台。它主要包含四类能力：第一是资料问答，也就是从资料中找证据并回答；第二是关系追踪，也就是追踪部件、故障、工况和维修动作之间的关系；第三是社区摘要，也就是把关系密切的一组节点综合起来；第四是全局综合，也就是回答跨文档、跨主题的综合性问题。

（停顿，切到主线页）

### 2. 汇报主线：不是一道问答题，而是一堆资料

这次汇报我想按照真实工程路线来讲，而不是只展示最后页面。因为这个项目最开始面对的不是一道简单问答题，而是一堆混乱的资料。

如果只是问大模型“联合循环为什么效率更高”，它可能能根据通用知识答出一个看起来正确的答案。但是我们要解决的问题不是这个。我们要解决的是：当资料来自 PDF、扫描页、JSON 标注、维修报告、教材和图表时，系统能不能把这些资料规整进去，能不能从里面检索到证据，能不能把答案和来源对应起来，最后还能不能通过评测证明它确实在变好。

所以我的汇报会沿着一条工程线展开：第一，资料入口，解决不同资料能不能进入同一系统；第二，证据构建，解决资料入库后如何变成可引用证据；第三，检索问答，解决用户提问时证据怎么被找到；第四，GraphRAG，解决单段证据不够时，如何追踪关系和做全局综合；第五，评测闭环，解决我们怎么证明系统不是只会演示，而是能评分、能归因、能继续优化。

（停顿）

### 3. 项目起点：早期只是让资料能进来

这个项目并不是一开始就长成现在这样。最早的时候，我跟着老师做这个 RAG 项目，目标其实很朴素，就是先做一个能够操作的控制台和页面。

当时我们主要做的是 JSON 解析，把标注数据解析成文本块，然后写入 ChromaDB 向量库。系统能够做查询、统计、导出，也能证明一件事：资料不是停留在文件夹里，而是可以进入我们的系统。

但是这只是第一步。早期页面上有上传、OCR、集合管理、搜索、图谱、导出、评测这些入口，看起来功能很多，但后来我意识到，能操作不等于可信，功能入口多也不等于系统完整。真正重要的是：资料进来以后，它能不能被正确处理，能不能被检索，能不能形成证据，能不能支撑答案。

所以这个项目的一个转折点，就是从“页面能不能跑起来”，转向“资料能不能被系统处理和验证”。

### 4. 真实资料倒逼数据入口：JSON、PDF 和 OCR

真实的动力装备资料并不是干净文本。这里面有文字型 PDF，有扫描型 PDF，有 JSON 标注，也有维修报告和教材图表。如果把它们都当成普通文本直接丢进 RAG，后面检索一定会出问题。

所以我们先把资料分成不同通道。JSON 本身有结构，里面可能已经包含页码、bbox、文本块、标签和来源，所以它适合作为第一批验证数据。文字型 PDF 可以直接抽取文本，但仍然要保留页码和来源。扫描型 PDF 本质上是图片，必须先经过 OCR，把图像变成文字。

这里 OCR 是我前期重点做的一件事。因为有些资料是图片型 PDF，如果没有 OCR，RAG 根本看不到里面的文字。我们试过不同路线，比如 Tesseract、PaddleOCR、RapidOCR，以及云 OCR。这里我想强调的不是“我调了很多 OCR 引擎”，而是：真实资料没有这么干净，RAG 的上限首先受资料质量影响。

开源 OCR 方案在中文扫描件上可能会出现错字、漏字、表格结构混乱；本地 OCR 服务可控，但需要处理启动、并发和环境；云 OCR 准确率更高，但会受到 API、网络和成本影响。最后我们形成的认识是：OCR 不是边角功能，它是扫描资料进入 RAG 的前置条件。没有这一步，后面的检索、图谱和评测都是空的。

（停顿，切到分块）

### 5. 证据构建：把厚资料切成可检索证据

资料进入系统以后，还不能直接回答问题。一本厚书、一份维修报告，必须被切成系统能检索的证据单元。

这里的分块，也就是 chunk，不是简单地每 500 个字切一刀。不同切法会带来不同问题。按页切，能保留页码，但有时候一页内容太多，主题会混在一起；按固定长度切，比较稳定，但可能把一个段落中间切断；按段落切，更符合语义，但不同段落长短差异很大；如果结合标题、段落断点、页码、来源和 overlap，就能更好地兼顾语义完整性和检索稳定性。

所以我们最终想要的不是一堆散乱文本，而是带有来源的证据块。每个证据块都应该知道：它来自哪个 PDF，在哪一页，属于哪个段落或标题，和前后文有什么关系。这样系统回答时，才不只是生成一句话，而是能把答案落回到证据页。

举一个例子，如果问题是“燃气-蒸汽联合循环为什么效率更高”，系统不能只回答“因为效率更高”。它应该检索到“燃气轮机排气余热”“余热锅炉”“蒸汽轮机继续做功”“能量梯级利用”这些证据块，并且能说明它们来自哪份资料、哪一页。这样答案才有可信基础。

### 6. 索引与检索：不是只放进向量库

证据块形成以后，下一步是索引。这里我想特别说明一下“向量”的含义。我们平时在数学里说向量，可能想到二维或三维坐标，但在 RAG 里，向量通常是模型给文本生成的高维语义坐标。每一个维度不是人工命名的“温度、压力、功率”，而是模型训练出来的隐含语义特征。

这些证据块进入 ChromaDB 后，就可以用语义相似度召回。但动力装备资料有一个特点：部件名、型号、故障名、工况术语非常明确。比如压气机、燃烧室、涡轮、余热锅炉、M701F，这些词本身就是很强的检索信号。

所以我们没有只做向量检索，而是比较了关键词检索、语义向量检索和混合检索。关键词检索类似 BM25，适合强术语；语义向量适合同义表达和隐含语义；混合检索用 RRF 之类的方法把多路召回融合起来。

这里的关键不是“方法越复杂越好”，而是要通过评测看哪种方法在当前资料上更有效。

（停顿，切到评测对比）

### 7. 检索评测：复杂方法不一定最高分

我们用同一批 60 道评测题比较了几种检索方式。结果很有意思：keyword 的 question recall@K 是 0.833，hybrid_rrf 是 0.817，dense_hashing 是 0.583。

这个结果说明，当前动力装备资料里强术语非常明显，所以关键词检索反而表现最好。混合检索接近关键词检索，但没有稳定超过它；语义向量不是没有价值，而是当前的 dense_hashing 只是一个可复现的离线语义基线，并不代表最终 embedding 的上限。

这给我们的启发是，不能盲目相信更复杂的方法。后续更合理的方向不是把所有方法无脑混合，而是先做问题路由。也就是说，用户提问后，系统先判断问题类型：如果是事实型、定义型、片段型问题，就优先走标准 RAG 和强术语检索；如果是关系追踪、跨文档综合、全局摘要类问题，就进入 GraphRAG 或社区摘要路线。

所以这页我想表达的是：评测让我们从主观演示走向了客观比较。不是看一次回答好不好，而是看同一批问题、同一套指标下，不同方法到底表现如何。

### 8. GraphRAG：不是替代所有 RAG，而是增强关系型问题

接下来讲 GraphRAG。这里我会比较谨慎，因为 GraphRAG 不能被说成替代所有 RAG 的万能路线。

标准 RAG 擅长的是从文本片段中找直接证据，比如某个定义、某个说明、某个操作步骤。但动力装备资料里还有很多问题不是单段证据能解决的。比如一个故障和哪些部件有关，一个工况变化会影响哪些指标，某类维修动作和哪些设备对象相连，这些问题更适合用实体关系来组织。

GraphRAG 的作用就是把文本中的实体、关系和证据连接起来。这里的节点可以是压气机、燃烧室、涡轮、余热锅炉、故障、工况、指标；边可以表示部件关系、影响关系、原因关系、维修动作关系。更重要的是，每条关系都不能凭空生成，而是要绑定 evidence，也就是能回到原文证据页。

所以我对 GraphRAG 的定位是：它是一条增强路线，不是替代路线。它适合关系追踪、社区摘要和全局综合问题。比如若干个实体关系很密切，可以形成一个社区，系统先对社区内部内容做摘要；多个社区之间再做全局综合，用来回答跨文档、跨对象的问题。

这也是为什么我们需要问题路由。不是每个问题都要 GraphRAG，也不是每个问题都只走普通 RAG。系统应该根据问题类型选择合适入口。

### 9. 系统分层：页面只是入口，能力在后面

随着项目推进，我们也把系统从一个页面逐渐拆成工程化分层。页面只是入口，真正的能力在后面的数据管线、检索层、图谱层、编排层、评测层和存储层。

数据管线负责资料解析、OCR、清洗和分块；检索层负责关键词、向量和混合召回；图谱层负责实体关系、schema 和 evidence 绑定；编排层负责把问题、证据和答案组织起来；评测层负责用统一问题集比较结果；存储层负责 ChromaDB、图谱数据和运行状态。

这个分层的意义是让项目从“能打开的页面”变成“可维护的系统”。早期页面能跑起来很重要，但如果所有能力都堆在页面上，后续就很难评测、很难修正、也很难解释。因此后面我们一直在把能力收回到系统结构里。

### 10. 评测平台：让项目从能演示变成能评分

最后是评测平台。这个部分对我来说很关键，因为它让项目从“我演示给你看”变成“系统自己能评分”。

我们在 GitHub Pages 上做了评测入口。它的目标不是简单展示评测结果，而是让一个 RAG 项目导入后，能够系统性地检查工程结构、评测集、baseline、失败归因、GraphRAG 覆盖和复现质量，并给出一个分数。

也就是说，评测平台不是只看最终答案，而是看这个项目有没有完整流程：资料是否能进入系统，证据是否能被构建，检索是否有 baseline，对失败是否有归因，GraphRAG 是否有明确边界，结果是否能被复现。

这里我还想强调一点：失败不是扣分项的结束，而是下一轮优化的起点。比如有的失败来自召回范围不够，有的来自排序没有把关键证据排到前面，有的来自混合检索稀释了关键词优势，有的来自评测概念没有覆盖完整。把这些失败分清楚以后，就能对应到下一步任务：扩展 source scope，引入 reranker，更换正式 embedding，或者对 GraphRAG 子集做 local/global 同题实测。

所以评测闭环的意义是：项目不是一次性做完，而是可以持续迭代。

（停顿，进入总结）

### 11. 总结：最后回到初心

最后，我想把这个项目收回到最初的问题：我们到底为什么要做这个系统？

我认为它的初心不是做一个看起来很复杂的网页，也不是堆很多技术名词，而是让动力装备资料从混乱的文件，变成可以被检索、可以被引用、可以被评测的可信答案。

这一路上我们其实走过一些弯路。早期我们更关注页面和功能入口，后来发现功能堆叠不能证明系统可信；中间我们花了很多精力在 OCR 上，后来意识到 OCR 本质上是资料进入 RAG 的前置条件；GraphRAG 也不能只靠画图展示，而必须绑定 evidence、明确适用问题；评测也不能只包装成果，而要真正回到 baseline、失败归因和优化路线。

所以今天我想汇报的不是“我们最后做了一个网页”，而是“我们怎样从一个能操作的控制台，逐步收敛成一个面向动力装备资料的可信问答与评测系统”。

它和普通 RAG demo 的区别在于：我们面对的是真实资料，不是干净文本；我们做了资料入口和 OCR，不是直接问答；我们比较了关键词、向量和混合检索，不是凭感觉选方法；我们引入 GraphRAG 处理关系型和全局型问题，但没有夸大它；我们做了 60 题评测和失败归因，让系统能够被评分、被解释、被继续优化。

如果下一阶段继续做，我会优先推进四件事：第一，换正式 embedding，提高语义召回能力；第二，引入 reranker，解决召回后排序不稳的问题；第三，对 GraphRAG local 和 global 做同题实测，明确它在哪些问题上真正超过普通 RAG；第四，把评测平台和任务队列进一步做稳定，让它从课程项目继续走向可复现实验平台。

我的汇报到这里，谢谢各位老师。

1. PowerRAG 不是让模型直接答，而是用检索增强生成答案的可信度。
2. 我们面对的不是一道问答题，而是一堆 PDF、扫描件、JSON 和维修报告。
3. 早期目标只是让资料能进系统，后来发现能操作不等于可信。

10. 页面只是入口，真正价值在数据、检索、图谱、编排、评测的工程分层。
11. 评测平台让项目从能演示变成能评分、能归因、能优化。
12. 最后回到初心：让动力装备资料变成可检索、可引用、可评测的可信答案。

## 答辩追问备用回答

### Q1：这个和直接问 ChatGPT 有什么区别？

区别在于资料来源和可追溯性。直接问 ChatGPT，答案主要来自模型已有知识；我们这里是先把动力装备资料处理成可检索证据，再让模型基于证据回答。答案旁边要能回到页码、来源和证据块，所以它更适合私有资料和专业资料场景。

### Q2：为什么 keyword 比 hybrid 还高？

这正是评测的价值。动力装备资料里有很多明确术语，比如部件名、型号、故障名和工况名，所以关键词检索本身就很强。hybrid 接近 keyword，但没有稳定超过它，说明下一步不能盲目融合，而要做问题路由、重排器和更正式的 embedding。

### Q3：GraphRAG 到底完成了什么？

当前完成的是构建链路和评测准备，包括 schema 约束、实体关系抽取、evidence 绑定、人工复核、图谱子集识别和边界判断。我不会说 GraphRAG 已经全面超过普通 RAG，更准确的说法是：它适合关系追踪、社区摘要和全局综合问题，后续要做 local/global 同题实测。

### Q4：评测系统的意义是什么？

它把项目从“能跑”变成“可比较”。同一批 60 道问题可以比较 keyword、dense、hybrid 的召回和覆盖，也能把失败分成召回不足、排序问题、概念覆盖不足等类型。这样失败就不是一句“效果不好”，而是下一轮优化任务。

### Q5：如果现场 GitHub API 403 怎么办？

这是 GitHub 匿名 API 的限流问题，不代表页面逻辑失败。评测页面已经做了内置快照降级，遇到 403 时可以继续展示评测流程和报告结果。现场不需要在这个问题上停太久，直接说明限流原因并切换到快照演示即可。

## 不能乱说的边界

- 不要说 GraphRAG 已经全面替代普通 RAG。
- 不要说 dense_hashing 代表最终 embedding 上限。
- 不要说 60 题评测就是最终权威 benchmark。
- 不要把网页说成系统全部，网页只是展示和评测入口。
- 不要只讲成果，要讲探索过程和每次修正为什么发生。
